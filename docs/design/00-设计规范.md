# EvalScope — Web 可观测平台 交互 UI 设计规范

> 版本 v1.0 · 2026-06 · 适用于 `web/` 多租户评估可观测平台前端重构
> 配套高保真原型见同目录 HTML 文件（用浏览器打开 `_index.html` 进入）

---

## 1. 设计目标与问题回应

本次设计针对反馈的两个核心问题：

| 反馈问题 | 设计对策 |
|---------|---------|
| **① 交互不符合行业最佳实践**，与 langfuse 差距大（创建项目 / 配置 API Key / 查看项目数据 / 运行详情 / 报告摘要 / 评估细则 / 原始文档预览） | 重构信息架构为 **组织 → 项目 → 运行 → 样本 → 约束** 五级钻取（drill-down）层级；每一级都有清晰的概览→明细路径；API Key 接入做成「签发即给接入代码片段」的引导式流程；文档预览改为 **左右分栏（结论 ↔ 原文）** 对照视图。 |
| **② 视觉难看**，缺风格 / 配色 / Logo | 建立完整设计系统（暗色仪器风 / 统一色板 / Inter + JetBrains Mono 双字体 / EvalScope reticle logo）；所有数据用等宽字体对齐，强化「测量仪表」观感。 |

### 品牌定位

- **品牌名**：EvalScope
- **一句话**：Agent 能力的「示波器」—— 把每一次评估运行的信号、约束与证据，量化、对照、可追溯。
- **调性**：科技 / 冷静 / 仪器级（instrument-grade）。克制的动效，数据优先，等宽字体承载所有量化信息。刻意避免渐变滥用、emoji、圆角卡通感等「AI 生成味」。
- **Logo 含义**：圆环 = 镜头/示波范围（scope），十字刻度 = 测量基准（reticle），上扬折线锁定到青色靶点 = 评估信号收敛到目标。

## 2. 信息架构（IA）

```
EvalScope
│
├─ 公开区
│   ├─ 落地页 index.html
│   ├─ 登录 login.html
│   └─ 注册 register.html（注册即创建/加入组织）
│
└─ 应用区（左侧固定导航 = 组织上下文，结构固定不变）
    ├─ 主导航（固定）：项目看板 / 全部运行
    ├─ 组织分区（固定）：成员 / 组织设置
    │   ├─ 成员 members.html          ← 成员列表 / 角色 / 邀请 / 待接受邀请
    │   └─ 组织设置 org-settings.html  ← 通用信息 / 套餐用量 / 危险区（转移所有权·删除组织）
    │
    ├─ 项目看板 dashboard.html        ← 组织下所有项目卡片 + 关键指标
    │   └─ 新建项目（Modal）
    │
    ├─ 项目详情 project-detail.html    ← 单项目
    │   ├─ 概览 Tab：趋势图 + 运行列表（指标含 ? 说明）
    │   ├─ 运行 Tab：可筛选的运行表
    │   └─ 设置 Tab（左侧子导航四面板）：
    │       ├─ API Key：新建 / 删除（删除即移除，无残留）+ 一次性密钥 + 接入代码
    │       ├─ 基本信息：名称 / slug / 描述 / 默认规则集
    │       ├─ 数据保留：运行样本保留期 + 制品保留期
    │       └─ 危险区：归档项目 / 删除项目（输入 slug 确认）
    │
    ├─ 运行详情 run-detail.html        ← 单次评估运行
    │   ├─ 指标卡（DR / CPR / Reward / CondR，含阈值对照 + ? 指标说明）
    │   ├─ 失败分布（按约束）
    │   ├─ 报告摘要（Markdown 渲染 + 下载）
    │   └─ 样本列表（可按状态/分数筛选、排序）
    │
    └─ 样本详情 sample-detail.html     ← 单个被评估文档（左右分栏）
        ├─ 左：分阶段约束结论（format / commonsense / quality）
        │      阶段得分 S_format/S_common/S_soft/S_pref 均带 ? 说明
        └─ 右：原始文档 / 渲染截图 预览（Tab 切换 + 同步定位）
```

### 导航与面包屑

- 左侧边栏结构**固定不变**：`主导航（项目看板 / 全部运行）` + `组织分区（成员 / 组织设置）`。
- **废弃随页面变化的「当前项目」动态子菜单**——旧版在不同页切换会改变侧栏内容，易让用户对导航产生不确定感。项目级操作统一收纳进「项目详情」页内的 Tab/子导航。
- 顶栏始终显示**面包屑**：`组织 / 项目 / 运行 #id / 样本 task_id`，任意一级可点击回跳。
- 钻取关系唯一且可逆，杜绝「迷路」。

### 指标可解释性（? 说明组件）

- 非直观指标（DR / CPR / Reward / CondR / S_format / S_common / S_soft / S_pref）旁配 `?` 按钮，点击弹出说明卡，含**定义 / 计算方法 / 评估标准**三段式。
- 组件：`.explain` + `.explain-btn` + `.explain-pop`（theme.css），交互在 `assets/ui.js`（点击切换、点击外部/Esc 关闭、同时仅开一个）。

## 3. 核心用户流程

### 流程 A：首次接入（onboarding）
```
注册账号 → 自动创建默认组织 → 新建项目（名称+slug）
   → 进入项目设置 → 签发 API Key
   → 复制接入代码片段（环境变量 + ResultSink 配置）
   → 评估器侧运行 → 数据经 HMAC 摄取 → 看板出现首个运行
```
关键体验点：**签发 Key 的弹窗直接给出可复制的接入代码**，而不是只丢一串密钥让用户自己查文档。

### 流程 B：日常巡检（最高频）
```
项目看板（一眼看哪个项目指标掉了）
   → 项目详情看趋势（哪次运行回归）
   → 运行详情看失败分布（哪类约束失败最多）
   → 样本详情看具体结论 + 对照原文（为什么失败）
```
每一步都从「聚合」自然下钻到「个体」，符合 overview-first / details-on-demand 原则。

### 流程 C：定位一次失败
```
运行详情 → 失败分布点击某约束（如 format.document_count）
   → 自动筛出该约束失败的样本
   → 进入样本 → 该约束已高亮展开 → 右侧定位原文证据
```

## 4. 设计系统（Design Tokens）

完整实现见 `assets/theme.css`。核心 token：

### 4.1 色板

| 角色 | 变量 | 值 | 用途 |
|------|------|----|----|
| 背景（底） | `--bg` | `#0a0c10` | 页面底色 |
| 背景（抬升） | `--bg-elevated` | `#0f1116` | 卡片 |
| 背景（表面） | `--bg-surface` | `#161922` | 输入框 / 表头 |
| 描边 | `--border` | `#20242e` | 分隔线 |
| 主文字 | `--text-primary` | `#eef1f6` | |
| 次文字 | `--text-secondary` | `#8b919e` | |
| 三级文字 | `--text-tertiary` | `#565c6a` | 元信息 |
| **品牌主色** | `--accent` | `#3d6dff` | 电光蓝，操作/选中 |
| **信号色** | `--signal` | `#2fe6c8` | 青，数据可视化第二色 |
| 成功 | `--success` | `#3fb950` | PASS |
| 警告 | `--warning` | `#d29922` | soft tier |
| 危险 | `--danger` | `#f85149` | FAIL / hard tier |
| 信息 | `--info` | `#58a6ff` | preference tier |

> 配色原则：底色冷而深，仅用主色 + 信号色两种品牌色，语义色仅在状态处出现。避免大面积彩色，量化数据靠等宽字体而非颜色取胜。

### 4.2 约束层级（Tier）语义色

| Tier | 含义 | 颜色 | 失败影响 |
|------|------|------|---------|
| `hard_gate` | 格式硬门禁，失败即 fail-fast，后续阶段 SKIP | 红 `--tier-hard` | `S_format = -3` |
| `hard_score` | 常识硬约束，不中断但标记 gate 未过 | 红 `--tier-hard` | `S_common = 0` |
| `soft` | 软约束，加权平均 | 黄 `--tier-soft` | 计入 `S_soft` |
| `preference` | 偏好约束，加权平均 | 蓝 `--tier-pref` | 计入 `S_pref` |

### 4.3 字体

- **界面字体**：Inter（400/500/600/700/800）—— 字重区分层级。
- **数据字体**：JetBrains Mono —— 所有数字、ID、指标、代码。等宽对齐是「仪器感」的关键。
- 字号阶：11(元信息) / 12 / 13(正文) / 13.5(输入) / 15(logo) / 22(页标题) / 28(指标值)。

### 4.4 圆角 / 阴影 / 动效
- 圆角：4/6/8/12/16/20px（卡片 12，弹窗 16）。
- 阴影仅用于浮层（弹窗 / 下拉 / hover 抬升），平面区域无阴影。
- 动效统一 `cubic-bezier(0.4,0,0.2,1)`，时长 120–150ms，仅做 hover/选中/展开，不做装饰性动画。
- **入场编排**：容器加 `.reveal`、子块加 `.r-1…r-6`，首屏按 70ms 步距错落淡入上浮（`es-rise`）。一次性、克制，是「精心设计」的最高性价比信号；`prefers-reduced-motion` 下自动关闭。
- **Live 信号**：`running` 状态点用 `.pulse`（`es-pulse` 1.6s 呼吸 + 光晕），呼应「实时摄取」。
- 氛围层（三层，z-index:0，全部 `pointer-events:none`）：
  - `.ambient` 顶部蓝光 + 右下青光径向渐变；
  - `.grid-overlay` 衰减网格（仪表台）；
  - `.scanlines` 极低强度 CRT 扫描线（`overlay` 混合），把「Scope/示波」概念落到每一个界面表面，而非仅停留在 logo。

### 4.5 数字排版（仪器读数细节）
- 所有等宽数字（`.mono / .num / .metric-value / 表格数字列 / 分数`）统一启用 `font-variant-numeric: tabular-nums` + `"tnum"/"zero"`，保证逐列对齐、刷新不跳动——这是「测量仪表」观感成立的关键细节。
- 指标卡 `hover` 时顶边点亮一道信号高光（`--accent-line` 横向渐变），暗示读数「通电」。

## 6. 各页面设计要点

### 6.1 落地页 `index.html`
Hero（品牌主张 + reticle 视觉）→ 能力特性（评估引擎 / 多租户摄取 / 证据可追溯 / 报告）→ 工作流三步（接入→运行→洞察）→ 产品预览（仿真看板）→ CTA。

### 6.2 登录 / 注册 `login.html` `register.html`
居中卡片 + 氛围光。登录支持密码；注册引导填写「组织名称」，落地即有租户上下文。错误态、一次性提示均有样式。

### 6.3 项目看板 `dashboard.html`
组织下项目卡片网格。每卡：项目名 + slug + 最新运行的 DR/Reward + 运行数 + 迷你趋势火花线 + 最近运行时间 + 状态。空态引导新建。右上「新建项目」弹窗。

### 6.4 项目详情 `project-detail.html`
- **概览**：DR/CPR/Reward/CondR 趋势折线（可切指标/周期）+ 最近运行列表。
- **运行**：完整运行表，按模式/规则集版本/时间筛选，分页。
- **设置**：基本信息编辑；**API Key 管理**（列表含公钥/调用次数/最近使用/状态/吊销；签发弹窗含一次性密钥 + 接入代码片段 + 环境变量说明）；数据保留天数；归档（危险区）。

### 6.5 运行详情 `run-detail.html`
- 顶部：运行号 + 状态 + 模式 + 规则集版本 + Langfuse 链接 + 下载报告。
- 四张指标卡含**阈值对照**（gauge 标出阈值线，达标/未达标着色）。
- **失败分布**：按约束的失败次数条形榜，可点击下钻筛样本。
- **报告摘要**：Markdown 渲染区 + 下载 JSON/MD。
- **样本表**：task_id / 状态 / Reward / 各阶段分数，可筛可排序，点击进样本。

### 6.6 样本详情 `sample-detail.html`（核心创新）
**左右分栏对照**：
- 左栏：按阶段分组（格式 / 常识 / 质量）的约束结论。每条：tier + PASS/FAIL + 名称 + 分数 + 原因；展开看 details/moduleResults/Judge 出处。失败项默认展开并锚点高亮。
- 右栏：原始文档预览（HTML iframe / Markdown / 文本）+ 渲染截图 Tab；顶部可切换制品。
- 设计意图：**「结论」与「证据」并置**，让用户无需来回跳转即可判断结论是否成立——这是评估平台区别于普通 BI 的关键。

## 7. 可访问性与响应式
- 对比度：主文字/背景 ≥ 12:1，次文字 ≥ 4.5:1。
- 所有可点元素有 hover/focus 态；表单 focus 有 3px 主色光圈。
- ≤920px 收起侧边栏（移动端以查看为主，编辑操作降级）。
- 颜色不作为唯一信息载体：PASS/FAIL 同时有文字与图标。

## 8. 落地建议（与现有技术栈对接）
- 现有前端为 React + Ant Design + ECharts。本规范的 token 可直接映射为 AntD `ConfigProvider` 的 `theme.token`（`colorPrimary=#3d6dff`、`colorBgBase=#0a0c10`、`borderRadius=8`、`fontFamily`）。
- 等宽数字：为指标/表格数字列统一加 `font-family: JetBrains Mono`。
- 趋势/分布图：ECharts 暗色主题，line 用 `--accent`/`--signal`，去网格线弱化。
- 原型 HTML 为**视觉与交互的事实标准**，研发以像素与行为对齐为准。
