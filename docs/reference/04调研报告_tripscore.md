# TripScore 评估体系详解

> 本文档基于论文《TripScore: Benchmarking and Rewarding Real-World Travel Planning with Fine-Grained Evaluation》(arXiv:2510.09011v3) 整理，系统介绍 TripScore 的评估框架、指标体系、约束定义、奖励计算方式及数据集构建方法。

---

## 一、评估框架概述

TripScore 采用 **"约束检查 → 分数聚合"** 的两层评估架构：

```
用户查询 (Query) + 相关信息 (Recalled Info)
           ↓
    规划模型生成行程计划 (Delivery Plan)
           ↓
    评估器逐条检查约束条件 (LLM + Rule-based)
           ↓
    输出最终奖励分数 (Final Score)
```

**核心设计原则**：评估过程中不直接给模型提供工具调用能力，而是将预先收集好的相关信息直接提供给规划模型。这样可以 **纯粹地评估模型的规划推理能力**，避免外部工具准确性对评估结果的干扰。

---

## 二、评估指标体系

论文使用以下 **5 个核心指标** 全面衡量模型性能：

| 指标 | 全称 | 含义 | 性质 |
|------|------|------|------|
| **DR** | Delivery Rate | 交付率：成功满足 **所有格式约束** 的计划比例 | 硬性门控 |
| **CPR** | Commonsense Constraint Pass Rate | 常识约束通过率：满足 **所有常识约束** 的计划比例 | 硬性门控 |
| **Reward** | Reward Score | 综合奖励分数：整合全部约束的细粒度质量分 | 连续评分 |
| **CondR** | Conditional Reward | 条件奖励：仅在通过 DR 和 CPR 的计划上计算的 Reward | 条件评分 |
| **Time** | Inference Time | 平均推理时间（秒）| 效率指标 |

**指标说明**：
- **DR 和 CPR** 是硬性门控（Hard Gate）：若计划未通过，则被视为不可行
- **Reward** 是连续质量分：即使计划可行，质量也有高低之分
- **CondR** 用于避免 DR 指标过高时压制 Reward 的区分度，提供更精细的可行计划间的比较
- **Time** 用于评估不同方法的推理效率

## 三、四类约束的详细评估方式

TripScore 将评估约束划分为 **4 大类、23 个细项**。每项约束采用 **Rule（规则自动判断）** 或 **LLM（大模型判断）** 进行评估：

> **Rule vs LLM 的选择原则**：逻辑/算术可明确判定的使用 Rule（如时间冲突、地点一致性）；需要语义理解或开放判断的使用 LLM（如景点是否"著名"、用户请求是否被满足）。

### 3.1 Format Constraint（格式约束）— 4 项，全部 Rule-based

确保生成计划的结构完整性和信息准确性：

| 子项 | 评估内容 |
|------|----------|
| Response Format | 响应是否严格遵循要求的结构和组织格式 |
| Information Verification | 景点、交通、酒店是否全部来自提供的参考信息 |
| Information Accuracy | 名称、出发时间等细节是否与提供的信息完全匹配 |
| Information Relevance | 描述是否准确匹配目标景点，不混淆不同地点的信息 |

### 3.2 Commonsense Constraint（常识约束）— 7 项，全部 Rule-based

检验模型在旅行规划中的逻辑合理性和现实可行性：

| 子项 | 评估内容 |
|------|----------|
| Information Completeness | 是否包含必要信息（住宿、交通等），多日行程是否都有安排 |
| Chronological Order | 活动是否按时间先后顺序排列 |
| Location Consistency | 每天的活动是否安排在旅行者当天实际所在的城市 |
| Operating Hours | 景点访问时间是否在其营业时间内 |
| Travel Block-Out | 交通出行时段内是否不安排其他活动 |
| Transport Consistency | 城市间交通是否连贯、合理，不绕路或重复 |

### 3.3 Soft Constraint（软约束）— 7 项，Rule + LLM 混合

评估行程的质量和实用性（非强制，但高度影响用户体验）：

| 子项 | 评估方式 | 评估内容 |
|------|----------|----------|
| Schedule Density | Rule | 每天行程是否张弛有度，不过于松散也不过于紧张 |
| Hotel Consistency | Rule | 在同一城市是否使用同一家酒店，避免频繁换房 |
| Daytime Utilization | Rule | 是否充分利用白天时间，避免将行程安排在晚间开始 |
| Unique Attractions | Rule | 同一景点在整段行程中是否只出现一次 |
| Location Clustering | Rule | 地理位置相近的景点是否被安排在同一天，以减少交通时间 |
| Iconic Landmarks | **LLM** | 行程是否包含当地的著名景点和必看景点 |
| Attraction Diversity | **LLM** | 景点类型是否多样化，避免过度重复同类景点 |

### 3.4 Personal Preference Constraint（个人偏好约束）— 5 项，Rule + LLM 混合

评估行程是否符合用户的个性化需求：

| 子项 | 评估方式 | 评估内容 |
|------|----------|----------|
| Budget Preference | Rule | 是否符合用户的预算预期（高端 / 经济 / 性价比）|
| Pacing Preference | Rule | 是否符合用户期望的旅行节奏（轻松 / 适中 / 紧凑）|
| Attraction Prioritization | Rule | 是否优先包含用户指定的景点类型 |
| Physical Effort Preference | Rule | 体力强度是否与用户要求匹配（轻度 / 中度 / 高强度）|
| User Request Fulfillment | **LLM** | 是否满足用户在查询中提出的特定请求 |

## 四、奖励分数的计算方法

最终奖励分数通过以下加权公式计算：

$$
\mathcal{R}(S; \theta) = S_{\text{format}} + S_{\text{com}} + w_3 \frac{\sum_j w_{1,j} S_{\text{soft},j}}{\sum_j w_{1,j}} + w_4 \frac{\sum_k w_{2,k} S_{\text{pref},k}}{\sum_k w_{2,k}}
$$

### 4.1 符号说明

| 符号 | 含义 |
|------|------|
| $S_{\text{format}}$ | 格式约束得分：通过 = +1，失败 = -3 |
| $S_{\text{com}}$ | 常识约束得分：通过 = +1，失败 = -1 |
| $S_{\text{soft},j}$ | 第 $j$ 个软约束子项得分，归一化到 $[0, 1]$ |
| $S_{\text{pref},k}$ | 第 $k$ 个偏好约束子项得分，归一化到 $[0, 1]$ |
| $w_1, w_2, w_3, w_4$ | 可学习的权重参数 |

### 4.2 字典序门控机制

奖励计算采用严格的层级判定：

1. **第一层 — 格式约束**：若格式检查失败，$S_{\text{format}} = -3$，评估立即终止
2. **第二层 — 常识约束**：若常识检查失败，$S_{\text{com}} = 0$（无奖励）
3. **第三层 — 软约束与偏好约束**：格式和常识均通过后，软约束和偏好约束**累加**贡献最终奖励

### 4.3 权重参数值

权重通过 **203 位旅游专家** 的成对比较标注数据训练优化，采用网格搜索 + 5 折交叉验证确定。合成数据集和真实数据集使用**不同的偏好权重**：

| 参数 | 合成数据 | 真实数据 |
|------|----------|----------|
| Schedule Density | 0.70 | 0.70 |
| Hotel Consistency | 0.50 | 0.50 |
| Daytime Utilization | 0.40 | 0.40 |
| Unique Attractions | 0.20 | 0.20 |
| Location Clustering | 0.70 | 0.70 |
| Iconic Landmarks | 0.10 | 0.10 |
| Attraction Diversity | 0.20 | 0.20 |
| Attraction Prioritization | 0.20 | — |
| Pacing | 0.60 | — |
| Budget | 0.60 | — |
| Physical Effort | 0.60 | — |
| User Request | — | 1.00 |
| $w_3$ (软约束乘数) | 1.00 | 1.00 |
| $w_4$ (偏好乘数) | 0.10 | 1.40 |

> **注**：真实世界数据使用单一的 "User Request" 项替代了合成数据的多个细分偏好项，且偏好权重乘数 $w_4$ 显著提高（0.10 → 1.40），反映了真实请求中个性化满足的重要性。

## 五、数据集构建与获取

### 5.1 数据集规模与划分

| 数据集 | 类型 | 数量 | 覆盖城市 | 酒店 | 景点 |
|--------|------|------|----------|------|------|
| Training | 合成 | 3,493 | 781 | 7,830 | 9,788 |
| Validation | 合成 | 100 | 70 | 1,057 | 1,525 |
| Validation | 真实 | 58 | 41 | 638 | 1,313 |
| Test | 合成 | 1,000 | 358 | 4,459 | 5,452 |
| Test | 真实 | 219 | 122 | 1,894 | 2,057 |
| **总计** | — | **4,870** | **897** | **9,376** | **10,997** |

### 5.2 合成查询的构建方式

- **目的地采样**：从携程用户交互日志中选取 **Top 900** 高频请求目的地
- **天数生成**：为每个目的地附加日志中最常选择的 **5 个行程天数**
- **偏好注入**：从预定义偏好集合中随机抽取用户偏好（预算、节奏、景点类型、体力等）
- **结果**：生成的查询覆盖常见旅行模式，但带有随机化偏好以增加多样性

### 5.3 真实世界查询的构建方式

- **数据来源**：携程生产旅行应用的真实用户请求（已获得用户授权）
- **收集方式**：用户以**自由形式文本**描述旅行需求，系统返回定制行程
- **覆盖范围**：
  - 具体任务型（如参加会议）
  - 日常景点偏好型（如第一天城市游、第二天动物园）
  - 主题旅行型（如哈利波特主题之旅）
- **过滤规则**：排除 **10 词以下** 的过于简单请求

### 5.4 相关信息检索（Recalled Information）

- 使用携程专业运营团队维护的**生产级行业数据**
- 包含景点、酒店、交通的详细信息
- **时间相关数据**（营业时间、季节性）归一化为当地时间
- 在评估基准中，**直接提供**这些信息给规划模型，跳过实际工具调用阶段

### 5.5 质量控制流程

采用**三层过滤**确保数据质量：

```
原始查询
    ↓
第一层：自动过滤 — 排除检索信息为空的查询
    ↓
第二层：LLM 过滤 — 使用 Gemini-2.5-flash 剔除信息不一致的样本
    ↓
第三层：人工审核 — 标注员移除过于困难或无意义的请求
    ↓
最终数据集
```

## 六、评估验证（与人类判断的对齐）

为确保评估器的可靠性，论文进行了严格的人工验证实验。

### 6.1 专家标注实验

| 项目 | 详情 |
|------|------|
| 专家人数 | 203 位旅游专家 |
| 标注方式 | 成对比较（A 更好 / B 更好 / 都不满意）|
| 标注总量 | 1,468 对行程（从 3,000 条查询生成）|
| 每对评估人数 | 3 位专家独立标注 |
| 专家间一致率 | 成对 71.69%，三专家整体 59.00% |

### 6.2 与人类判断的对齐度

| 对比维度 | 结果 |
|----------|------|
| 原始一致率 | **60.75%** |
| 人类可靠性上限 | 83.9% |
| 模型达到人类上限比例 | **82.8%** |

### 6.3 TripScore vs LLM-as-judge

| 方法 | 最佳模型 | 与人类一致率 | Kendall's τ |
|------|----------|-------------|-------------|
| Point-wise | Gemini-2.5-pro | 62.35% | 0.2347 |
| Pair-wise | Gemini-2.5-pro | 58.77% | 0.1675 |
| **Ours (TripScore)** | **Gemini-2.5-flash** | **60.75%** | **0.1892** |

TripScore 相比 LLM-as-judge 的核心优势：
1. **成本低**：无需多次调用 LLM 进行成对比较
2. **可扩展性**：输出连续分数，可直接用于 RL 训练
3. **可解释性**：基于明确约束，能提供具体错误反馈

## 七、总结对照表

| 问题 | 答案 |
|------|------|
| **如何评估** | 基于 4 类 23 项约束对行程进行 Rule + LLM 混合检查，通过加权公式聚合为单一 Reward |
| **评估指标** | DR（交付率）、CPR（常识通过率）、Reward（综合奖励）、CondR（条件奖励）、Time（推理时间）|
| **如何获得指标** | 硬约束（格式/常识）使用 Rule 自动判定；软约束/偏好使用 Rule + LLM 混合判定；Reward 通过加权公式聚合 |
| **数据集获取** | 合成数据来自用户日志采样 + 随机偏好；真实数据来自携程生产环境的授权用户自由请求；经过三层质量控制 |
| **权重来源** | 通过 203 位旅游专家的成对比较标注数据，经网格搜索 + 交叉验证优化得到 |
| **与人类对齐度** | 原始一致率 60.75%，达到人类可靠性上限的 82.8% |
