# 开源 Agent 能力评估框架全景图

> 本文档基于 2024-2026 年行业最新资料，系统梳理当前主流的开源 Agent 评估框架、Benchmark 和工具库，并分析其适用场景与关键趋势。

---

## 一、顶级综合评估 Benchmark

以下 Benchmark 被业界公认为 Agent 能力的"试金石"，覆盖不同领域和评估维度。

### 1. AgentBench（清华大学，2023）

| 属性 | 说明 |
|------|------|
| **领域** | 多环境综合（8个异构环境） |
| **任务数** | ~1,091 |
| **测试能力** | OS操作、数据库、网页浏览、游戏、家务、购物等 |
| **评估方式** | 每环境自定义检查逻辑 |
| **前沿得分** | ~54.3% 综合（趋于饱和） |
| **特点** | 覆盖面最广的通用 Agent 基准，适合初步筛选模型潜力 |
| **局限** | 每个域深度有限；聚合分数可能掩盖单域缺陷 |

**环境列表**：操作系统、数据库、网页浏览、知识图谱、数字卡牌游戏、横向思维谜题、家务任务、网页购物

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 每个环境预定义任务集，包含初始状态、目标描述和期望结果；共 8 个异构环境各持有独立的任务定义格式 |
| **执行环境搭建** | 为 8 类环境分别搭建隔离沙箱：操作系统（Linux 容器）、数据库（SQLite/PostgreSQL 实例）、网页浏览（无头浏览器）、游戏环境（模拟器）等 |
| **Agent 接入方式** | Agent 通过标准化 API 与各环境交互，每个环境提供统一的 `step(action)` → `observation` 接口；Agent 不直接接触环境内部实现 |
| **过程监控** | 每一步的 action 和 observation 都被记录；统计交互轮数、工具调用次数、是否进入无效循环 |
| **产物收集** | 收集每个任务的最终状态快照 + 完整交互轨迹（action-observation 序列） |
| **结果判定** | 每个环境自定义检查逻辑：OS 环境检查文件系统状态，数据库环境检查 SQL 执行结果，购物环境检查购物车状态等 |

```
测试用例 → 注入隔离环境 → Agent 通过 API 交互（step by step）
                              ↓
                        记录 action-observation 序列
                              ↓
                    收集最终状态 + 交互轨迹 → 环境特定检查逻辑判定
```

### 2. SWE-bench（Princeton + OpenAI，2024）

| 属性 | 说明 |
|------|------|
| **领域** | 软件工程 |
| **任务数** | 500（Verified）/ 731（Pro） |
| **测试能力** | 真实 GitHub Issue 修复、Bug 修复 |
| **评估方式** | 单元测试通过 + 无回归 |
| **前沿得分** | 87.6%（Claude Opus 4.7） |
| **特点** | 真实世界代码任务，难以通过记忆作弊 |
| **变体** | SWE-bench Verified / Pro / Multimodal |

**适用场景**：代码 Agent、开发者工具、代码生成系统评估

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 从真实 GitHub 仓库提取 Issue + 对应的 PR/Merge Commit 作为金标准；每个用例包含：问题描述、仓库快照（base commit）、通过测试的金标准 patch |
| **执行环境搭建** | Docker 容器，内含完整的真实代码仓库（checkout 到 base commit）+ 依赖环境 + 测试套件（pytest/unittest） |
| **Agent 接入方式** | Agent 在容器内工作，可通过 bash 执行任意命令（浏览代码、编辑文件、运行测试）；SWE-bench 本身不限制 Agent 的实现方式，只提供任务输入和验证环境 |
| **过程监控** | 框架本身不主动监控 Agent 过程，由 Agent 自行记录（或由接入的 scaffold 如 OpenHands 记录）；仅关注最终生成的 patch |
| **产物收集** | 收集 Agent 生成的代码 patch（diff 格式） |
| **结果判定** | 将 patch 应用到仓库 → 运行与 Issue 相关的单元测试 → 检查：①相关测试通过 ②不引入回归（原有测试仍通过） |

```
GitHub Issue + 仓库快照
    ↓
Docker 容器启动（含真实仓库 + 测试套件）
    ↓
Agent 在容器内执行（浏览代码、编辑、运行测试）
    ↓
收集 Agent 生成的 patch
    ↓
应用 patch → 运行单元测试 → 通过/回归检查
```

### 3. WebArena（CMU，2023）

| 属性 | 说明 |
|------|------|
| **领域** | Web 导航与操作 |
| **任务数** | 812 |
| **测试能力** | 多步骤网页任务（电商、论坛、CMS、GitLab 克隆站） |
| **评估方式** | 页面状态检查 / API 检查 / LLM-as-judge |
| **前沿得分** | 68.7%（Claude Mythos Preview） |
| **人类基线** | ~78% |
| **特点** | 自托管真实网站克隆，动态内容交互 |

**关键挑战**：弹窗处理、视觉布局理解、UI 变化适应

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 每个任务由自然语言指令 + 初始 URL + 期望的最终状态描述构成；覆盖电商购物、论坛发帖、CMS 内容管理、GitLab 项目操作等多步骤场景 |
| **执行环境搭建** | 自托管 4 个真实网站克隆（电商 Magento、论坛 phpBB、CMS Content Management、GitLab），部署在隔离服务器上，提供真实的动态内容交互 |
| **Agent 接入方式** | Agent 通过浏览器自动化（如 Playwright/Selenium）操作网站；支持纯文本 DOM 观察和多模态（截图）观察两种模式 |
| **过程监控** | 记录每一步的浏览器操作（点击、输入、滚动、导航）；捕获页面 DOM 快照或截图作为 observation |
| **产物收集** | 收集最终页面状态（DOM/截图）+ 完整浏览器操作轨迹 |
| **结果判定** | 三种方式：①页面状态检查（元素存在/属性值）② API 检查（数据库/后端接口验证）③ LLM-as-judge（对最终页面截图进行语义判定） |

```
自然语言任务描述
    ↓
启动浏览器 → 导航到目标网站克隆
    ↓
Agent 通过浏览器自动化交互（点击、输入、导航）
    ↓ 每步记录
操作轨迹 + 页面快照/DOM 状态
    ↓
任务完成后检查最终页面状态
    ↓
页面元素检查 / API 调用验证 / LLM-as-judge 判定
```

### 4. OSWorld（XLang，NeurIPS 2024）

| 属性 | 说明 |
|------|------|
| **领域** | 桌面操作系统 |
| **任务数** | 369 |
| **测试能力** | 开放域任务（办公、系统工具、创意应用） |
| **评估方式** | 应用状态匹配 / 截图对比 |
| **前沿得分** | ~38-79%（因设置不同） |
| **平台** | Ubuntu / Windows / macOS |
| **特点** | 多模态（截图 + 动作），真实 OS 交互 |

**适用场景**：桌面自动化、Computer-Use Agent（如 OpenAI CUA、Anthropic Computer Use）

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 369 个任务覆盖办公软件（LibreOffice/Chrome）、系统工具（文件管理器、终端）、创意应用（GIMP/VLC）；每个任务定义初始桌面状态 + 目标描述 + 验证函数 |
| **执行环境搭建** | 启动真实的虚拟机（Ubuntu/Windows/macOS），预装全套桌面应用；支持多平台，每个任务在独立的 VM 快照上执行，保证环境可重置 |
| **Agent 接入方式** | 多模态交互：Agent 接收屏幕截图作为观察，输出鼠标/键盘操作（点击坐标、键入文本、快捷键）；支持 a11y（无障碍辅助）树作为补充观察 |
| **过程监控** | 每一步记录：①操作前截图 ② Agent 输出的动作 ③操作后截图；最多允许设定步数上限，超过视为失败 |
| **产物收集** | 收集最终应用状态（文件内容/应用数据）+ 完整操作轨迹（截图序列 + 动作序列） |
| **结果判定** | 应用状态匹配：通过预设的验证函数检查应用内部状态（如文件是否正确创建、表格数据是否正确填充）；截图对比用于视觉验证 |

```
任务描述 + 初始 VM 快照
    ↓
启动虚拟机（含真实桌面应用）
    ↓
Agent 通过截图观察 + 鼠标键盘操作交互
    ↓ 每步记录
操作前截图 → Agent 动作 → 操作后截图
    ↓
任务完成后执行验证函数
    ↓
检查应用内部状态（文件/数据/配置）是否达到预期
```

### 5. GAIA（Meta + Hugging Face，2023）

| 属性 | 说明 |
|------|------|
| **领域** | 通用助手 |
| **任务数** | 466 |
| **测试能力** | 真实世界问题解决（需网页、文件、代码、推理结合） |
| **评估方式** | 答案精确匹配 |
| **前沿得分** | 74.6%（Claude Sonnet 4.5 + HAL scaffold） |
| **难度分级** | Level 1/2/3（Level 3 裸模型 <35%，仍基本未解决） |
| **特点** | 对人类概念简单但对 AI 困难的综合任务 |

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 466 个需要多步推理的真实世界问题；每个用例包含：问题文本 + 附带文件（PDF/图片/代码等）+ 标准答案；分为 Level 1/2/3 三个难度 |
| **执行环境搭建** | 不提供固定执行环境；Agent 需要自行使用网页搜索、文件解析、代码执行等工具来解决问题；框架关注的是最终答案是否正确，而非执行路径 |
| **Agent 接入方式** | 开放式：不限制 Agent 如何解决问题，Agent 可使用任何工具组合（浏览网页、阅读文件、运行代码）；需要 Agent 自行搭建工具链 |
| **过程监控** | GAIA 基准本身不监控过程；仅记录提交的最终答案和推理时间；若使用 scaffold（如 HAL），则 scaffold 会记录工具调用轨迹 |
| **产物收集** | 仅收集最终答案（短文本/数值） |
| **结果判定** | 答案精确匹配：将 Agent 输出的最终答案与标注的标准答案对比，完全匹配即为正确 |

```
问题文本 + 附带文件（PDF/图片/代码等）
    ↓
Agent 自主执行（可使用网页搜索、文件解析、代码执行等工具）
    ↓
（GAIA 框架不控制执行过程）
    ↓
收集 Agent 提交的最终答案
    ↓
与标准答案精确匹配判定
```

### 6. τ-bench（Tau-Bench）

| 属性 | 说明 |
|------|------|
| **领域** | 多轮对话 + 策略遵循 |
| **测试能力** | 客服、预订、退款等多轮策略合规任务 |
| **评估方式** | 状态转换验证 + 策略规则检查 |
| **特点** | 强调多轮交互中的策略遵守，而非单轮正确性 |

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 领域特定的多轮对话场景，每个用例定义：用户角色（模拟客户）、对话策略规则（如退款政策、预订规则）、期望的状态转换路径 |
| **执行环境搭建** | 模拟的对话环境，包含用户模拟器（扮演客户提出请求）和策略引擎（定义业务规则） |
| **Agent 接入方式** | Agent 扮演客服角色，与用户模拟器进行多轮对话；每轮接收用户消息，输出回复动作 |
| **过程监控** | 逐轮记录对话内容、Agent 决策、是否遵循策略规则；检查每轮对话后的状态转换是否合规 |
| **产物收集** | 完整对话记录 + 每轮状态快照 |
| **结果判定** | 状态转换验证（对话过程中是否违反业务规则）+ 策略规则检查（最终结果是否符合政策要求） |

```
用户模拟器 + 策略规则定义
    ↓
Agent（客服角色）与用户模拟器多轮对话
    ↓ 每轮记录
对话内容 + Agent 决策 + 状态转换
    ↓
逐轮检查策略合规性 + 最终状态验证
```

## 二、2024-2025 新兴评估框架

### 1. MASEval（2025）⭐ 多 Agent 系统评估

| 属性 | 说明 |
|------|------|
| **定位** | 从单 Agent 模型评估扩展到完整多 Agent 系统评估 |
| **核心创新** | 解耦被测系统与评测框架，支持跨框架评估 |
| **能力** | 支持 per-agent tracing，无需锁定特定厂商实现 |
| **论文** | MASEval: Extending Multi-Agent Evaluation from Models to Systems |

**价值**：解决了多 Agent 系统中"黑盒化"评估的难题

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 从被测的多 Agent 系统中采集真实任务场景，解耦测试定义与系统实现；支持跨框架的任务描述格式 |
| **执行环境搭建** | 不搭建独立环境，直接对接被测的多 Agent 系统运行时；通过标准化接口（非特定厂商 API）接入 |
| **Agent 接入方式** | **被动观测模式**：MASEval 不控制 Agent 执行，而是在多 Agent 系统外部以"旁路"方式采集数据 |
| **过程监控** | 支持 **per-agent tracing**：为系统中每个 Agent 独立记录其决策链、工具调用、消息收发；不锁定特定厂商实现，通过通用 trace 格式适配 |
| **产物收集** | 每个 Agent 的 trace 数据 + 系统级交互拓扑（Agent 之间的消息流）+ 最终输出 |
| **结果判定** | 基于系统级行为分析，评估整体协作效果而非单个 Agent 的表现 |

```
多 Agent 系统运行
    ↓ （旁路观测）
MASEval 采集 per-agent trace
    ├── Agent A: 决策链 + 工具调用 + 消息收发
    ├── Agent B: 决策链 + 工具调用 + 消息收发
    └── Agent N: ...
    ↓
系统级交互拓扑分析
    ↓
评估整体协作效果 + 各 Agent 独立表现
```

### 2. Agent GPA Framework（2025）⭐ Agent 对齐评估

| 属性 | 说明 |
|------|------|
| **定位** | Goal-Plan-Action 对齐度评估 |
| **指标** | Goal Fulfillment / Plan Quality / Tool Selection / Plan Adherence / Tool Calling / Logical Consistency / Execution Efficiency |
| **方法** | 无参考（Reference-free）LLM-as-Judge 自动评估 |
| **验证** | 在 TRAIL/GAIA、TRAIL/SWE-bench 和生产数据分析 Agent 上验证 |

**价值**：提供了 Agent 内部思维链质量的结构化评估方法

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 复用已有 Benchmark（TRAIL/GAIA、TRAIL/SWE-bench）的任务定义，不自建测试集 |
| **执行环境搭建** | 不搭建环境；在已有 Benchmark 的执行环境中运行，Agent GPA 作为评估层叠加 |
| **Agent 接入方式** | **事后评估模式**：采集 Agent 已完成的执行 trace（包含 Goal → Plan → Action 链条），对 trace 进行结构化分析 |
| **过程监控** | 不实时监控；对已有的完整 trace 进行回溯分析，逐层拆解：Goal（目标理解）、Plan（规划质量）、Action（执行效率） |
| **产物收集** | 需要 Agent 输出结构化的执行 trace（目标设定、规划步骤、每步动作及结果） |
| **结果判定** | 使用 Reference-free LLM-as-Judge 对 7 个维度（Goal Fulfillment / Plan Quality / Tool Selection / Plan Adherence / Tool Calling / Logical Consistency / Execution Efficiency）逐一评分 |

```
Agent 执行 trace（Goal → Plan → Action 链条）
    ↓
结构化拆解为三层次
    ├── Goal 层：目标理解是否正确
    ├── Plan 层：规划是否合理高效
    └── Action 层：执行是否遵循计划
    ↓
LLM-as-Judge 对 7 个维度逐一评分（无需标准答案）
    ↓
输出各维度评分 + 对齐度分析
```

### 3. MCPEval（2025）⭐ MCP 标准化评估

| 属性 | 说明 |
|------|------|
| **定位** | 基于 Model Context Protocol (MCP) 的自动化深度评估 |
| **核心创新** | 跨领域端到端任务自动生成，消除手动构建评估管道的繁琐 |
| **特点** | 标准化指标、与原生 Agent 工具无缝集成 |

**价值**：代表了 Agent 评估向"协议化、自动化"演进的最新方向

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 基于 MCP 协议自动生成跨领域的端到端测试任务，消除手动构建评估管道的繁琐 |
| **执行环境搭建** | 利用 MCP 标准协议连接 Agent 与评估工具链，Agent 通过 MCP 接口调用评估工具；无需为每个领域定制环境 |
| **Agent 接入方式** | Agent 实现标准 MCP 接口即可接入；评估工具作为 MCP Server 提供标准化能力（数据注入、状态查询、结果采集） |
| **过程监控** | 通过 MCP 协议天然支持过程追踪；每次工具调用都是标准化的 MCP 请求/响应，可被评估框架透明捕获 |
| **产物收集** | MCP 协议化的标准输出格式，包含任务执行过程和结果的完整结构化数据 |
| **结果判定** | 标准化指标计算，与原生 Agent 工具无缝集成，支持跨领域统一的评估维度 |

```
MCP 协议自动生成测试任务
    ↓
Agent 通过标准 MCP 接口执行任务
    ↓ （每次工具调用通过 MCP 协议透明传输）
评估框架自动捕获调用轨迹
    ↓
标准化指标计算 + 结果采集
    ↓
跨领域统一评估报告
```

### 4. OpenHands（2025）

| 属性 | 说明 |
|------|------|
| **定位** | 开源代码 Agent 开发 + 评估平台 |
| **特点** | 事件驱动架构、迭代文件编辑、沙箱容器内执行 |
| **支持** | CodeAct 架构、BrowsingAgent、API 调用与浏览混合 |
| **Benchmark** | SWE-Bench、SWE-bench Verified |

**执行过程**：

| 环节 | 实现方式 |
|------|----------|
| **测试用例管理** | 对接 SWE-bench 任务集（真实 GitHub Issue + 仓库快照）；也可自定义任务 |
| **执行环境搭建** | **Docker 沙箱容器**：每个任务启动独立的 Docker 容器，内含完整代码仓库 + 运行环境；容器内可执行 bash 命令、编辑文件、运行测试 |
| **Agent 接入方式** | Agent 通过事件驱动架构与沙箱交互：发送动作（bash 命令/文件编辑/浏览器操作）→ 接收 observation（命令输出/文件内容/截图）；支持 CodeAct（代码即动作）和 BrowsingAgent（浏览器操作）两种模式 |
| **过程监控** | 事件驱动：每个动作和响应都是一个 Event，完整记录执行轨迹；支持迭代文件编辑（多轮修改同一个文件），每轮 diff 都被捕获 |
| **产物收集** | 代码 patch（diff 格式）+ 完整 Event 日志 + bash 执行输出 + 文件编辑历史 |
| **结果判定** | 基于 SWE-bench 的单元测试验证：应用 patch → 运行测试 → 通过/回归检查 |

```
SWE-bench 任务（Issue + 仓库快照）
    ↓
启动 Docker 沙箱容器
    ↓
Agent 通过事件驱动交互
    ├── bash 执行 → 捕获输出
    ├── 文件编辑 → 记录 diff
    └── 浏览器操作 → 截图观察
    ↓
收集代码 patch + Event 日志
    ↓
应用 patch → 运行单元测试 → 通过/回归检查
```

## 六、关键趋势（2024-2026）

### 6.1 评估方法演进

```
2023: 单任务 + 规则检查
    ↓
2024: 多环境 + LLM-as-judge
    ↓
2025: 级联评估 + 自动化生成 + MCP 协议化
    ↓
2026: 多 Agent 系统级评估 + 开放环境 + 行为分析
```

### 6.2 七大关键趋势

| 趋势 | 说明 |
|------|------|
| **1. 多 Agent 系统评估** | 从单 Agent 评估转向系统级评估（MASEval、GAMMAF） |
| **2. 开放环境测试** | 从封闭单任务转向真实、无约束场景（CyberExplorer、OSWorld） |
| **3. 评估协议化** | MCP 等标准化协议降低评估构建成本（MCPEval） |
| **4. 安全/对抗聚焦** | 红队测试、异常监控、攻防能力评估成为标配 |
| **5. 无参考评估** | LLM-as-Judge 减少对人类标注的依赖 |
| **6. 跨框架兼容** | 避免厂商锁定，支持公平对比（MASEval） |
| **7. 行为分析** | 不仅关注"是否成功"，更关注"如何失败"（Agent GPA、EnvTrustBench） |

### 6.3 严重警示：Benchmark 被"破解"问题 ⚠️

2026 年 Berkeley 研究显示，**所有主流 Benchmark 均可被 exploit 到接近满分**：

| Benchmark | 漏洞类型 | Exploit 得分 |
|-----------|----------|-------------|
| SWE-bench | Pytest hooks 强制通过 | ~100% |
| WebArena | 配置泄露 + DOM 注入 | ~95% |
| GAIA | 答案泄露 + 归一化冲突 | ~98% |
| OSWorld | VM 状态操纵 + 金标准文件 | ~90% |

**教训**：
1. **公开 Benchmark 只能设底线**，不能作为部署决策的唯一依据
2. **必须构建内部私有评估集**（held-out evaluation）
3. **多 Benchmark 交叉验证**，单一分数不足信
4. **关注过程指标**（步数、工具调用轨迹），而非仅关注最终结果

## 八、总结：从 Benchmark 到生产评估

```
┌─────────────────────────────────────────────────────────────┐
│                     Agent 评估层次金字塔                      │
├─────────────────────────────────────────────────────────────┤
│  第4层: 在线评估 (Online)                                     │

│  第3层: 私有评估 (Private)                                    │
│         内部 held-out 数据集 → 防止 benchmark 污染             │
│         └─ 部署决策的关键依据                                  │
├─────────────────────────────────────────────────────────────┤
│  第2层: 公开 Benchmark                                        │
│         AgentBench / SWE-bench / WebArena / GAIA / OSWorld    │
│         └─ 设底线、做对比，但不可全信                          │
├─────────────────────────────────────────────────────────────┤
│  第1层: 单元/模块评估 (Unit)                                  │
│         Rule-based + LLM-as-judge + 执行验证                   │
│         └─ 开发调试、快速迭代                                  │
└─────────────────────────────────────────────────────────────┘
```

**核心理念**：
1. **没有银弹**：单一 Benchmark 或方法无法全面评估 Agent
2. **分层评估**：开发期用单元评估 → 测试期用 Benchmark → 上线前用私有评估 → 上线后用在线评估
3. **过程 > 结果**：不仅关注 Agent 做对了没有，更关注它是怎么做的（步数、轨迹、工具选择）
4. **持续迭代**：评估体系需要随 Agent 能力进化而更新，防止评估失效

## 九、Agent 执行过程模式分类

综合以上所有框架的执行过程，可归纳为 **四种执行控制模式**：

### 9.1 四种模式总览

| 模式 | 核心特征 | 评估系统控制 Agent？ | 过程监控能力 | 代表框架 |
|------|----------|---------------------|-------------|----------|
| **A. 容器沙箱执行** | Agent 在隔离容器内执行，评估系统完全控制环境 | ✅ 完全控制 | ✅ 全量采集 | SWE-bench、OpenHands、AnalysisBench |
| **B. 模拟环境交互** | 搭建仿真环境，Agent 作为"玩家"操作 | ⚠️ 控制环境，不控制行为 | ✅ 动作轨迹 | AgentBench、WebArena、OSWorld、CyberExplorer |
| **C. 事后产出评估** | 仅评估最终输出，不参与 Agent 执行过程 | ❌ 不控制 | ❌ 仅看成品 | GAIA、τ-bench、AgentSelect |
| **D. 协议化旁路追踪** | 通过标准协议或旁路方式采集 Agent 内部状态 | ❌ 不控制 | ✅ 协议化 Trace | MASEval、MCPEval、Agent GPA、EnvTrustBench |

### 9.2 模式 A：容器沙箱执行

```
评估系统完全控制 Agent 的执行环境
┌──────────────────────────────────────────────────┐
│ 测试用例 → 注入隔离容器 → Agent 在容器内执行       │
│                              ↑ 过程全量采集        │
│                              ↑ 收集产物            │
│             自动化验证 ←── 执行结果（patch/报告）   │
└──────────────────────────────────────────────────┘
```

| 适用场景 | 代码生成、软件开发、程序分析等 Agent |
|----------|--------------------------------------|
| **优势** | 过程数据完整、环境可复现、结果判定自动化程度高 |
| **代价** | 需要为每个任务搭建 Docker 容器，资源消耗大；Agent 需适配沙箱接口 |
| **过程指标** | 命令执行序列、文件编辑 diff、测试运行结果 |

**代表框架对比**：

| 框架 | 容器类型 | 产物 | 判定方式 |
|------|----------|------|----------|
| SWE-bench | Docker（真实代码仓库） | 代码 patch | 单元测试通过 + 无回归 |
| OpenHands | Docker + 事件驱动 | patch + Event 日志 | SWE-bench 单元测试 |
| AnalysisBench | 构建环境（编译器 + 工具） | 分析报告 | 缺陷定位准确性 |

### 9.3 模式 B：模拟环境交互

```
评估系统搭建可控的仿真环境，Agent 在其中自由操作
┌──────────────────────────────────────────────────┐
│ 模拟网站 / 虚拟机 / 网络环境                       │
│         ↑ 截图/状态快照        ↓ Agent 操作动作    │
│    初始状态 ←────── Agent 通过原生接口交互 ──→ 最终状态│
│         ↑                                          │
│    每步记录动作轨迹                                 │
│         ↓                                          │
│    最终状态匹配 / 元素检查 / LLM 判定               │
└──────────────────────────────────────────────────┘
```

| 适用场景 | Web 自动化、桌面操作、安全攻防等需要与环境交互的 Agent |
|----------|-------------------------------------------------------|
| **优势** | 真实交互体验、验证 Agent 的实际操作能力 |
| **代价** | 环境搭建成本高（网站克隆、VM 部署、网络模拟）；环境维护复杂 |
| **过程指标** | 每步操作（点击/输入/命令）+ 每步观察（截图/DOM/输出） |

**代表框架对比**：

| 框架 | 环境类型 | 交互方式 | 判定方式 |
|------|----------|----------|----------|
| AgentBench | 8 类异构沙箱 | 标准化 `step(action)` API | 每环境自定义检查 |
| WebArena | 自托管网站克隆 | 浏览器自动化 | 页面状态/API/LLM 检查 |
| OSWorld | 真实虚拟机 | 截图 + 鼠标键盘操作 | 应用状态匹配 |
| CyberExplorer | 真实攻击模拟网络 | 自主网络操作 | Flag 获取 + 行为分析 |
| GAMMAF | 多 Agent 协作图 | Agent 间消息通信 | 图异常检测 |
| τ-bench | 模拟对话环境 | 多轮对话 | 策略规则合规检查 |

### 9.4 模式 C：事后产出评估

```
评估系统与 Agent 完全解耦，只评估最终输出
┌─────────────────┐     ┌──────────────────┐
│ Agent 执行（外部）│     │ 评估系统（独立）   │
│                 │     │                  │
│ 测试数据 → Agent │────→│ 最终输出 → 评估器  │
│ （不由评估系统   │     │ → 指标 → 报告     │
│   控制）         │     │                  │
└─────────────────┘     └──────────────────┘
```

| 适用场景 | 产出为文本/文件的 Agent，不关心执行过程 |
|----------|----------------------------------------|
| **优势** | 实现简单、与 Agent 完全解耦、对 Agent 无侵入 |
| **代价** | 无法采集过程指标（步数、工具调用、重试次数等） |
| **过程指标** | 仅推理时间或无 |

**代表框架对比**：

| 框架 | 输入 | 产出 | 判定方式 |
|------|------|------|----------|
| GAIA | 问题 + 附带文件 | 最终答案 | 精确匹配 |
| τ-bench | 多轮对话场景 | 对话记录 + 最终状态 | 策略规则检查 |
| AgentSelect | 异构评估报告 | 推荐排序 | 偏好信号转化 |

### 9.5 模式 D：协议化旁路追踪

```
通过标准协议或旁路方式，在不控制 Agent 的前提下采集内部状态
┌──────────────────────────────────────────────────┐
│ Agent 系统运行                                     │
│    │                                              │
│    ├── 标准 MCP 协议 ──→ 评估框架透明捕获调用轨迹    │
│    ├── 旁路 Trace 采集 ──→ per-agent tracing       │
│    └── 执行 trace 回传 ──→ 结构化分析              │
│                                                   │
│    ↓                                              │
│ 协议化的标准 Trace → 评估分析 → 报告                │
└──────────────────────────────────────────────────┘
```

| 适用场景 | 多 Agent 系统、内部过程评估、跨框架评估 |
|----------|----------------------------------------|
| **优势** | 不控制执行但有过程数据；跨框架通用；对 Agent 侵入性低 |
| **代价** | 需要 Agent 支持 trace 输出或协议接入 |
| **过程指标** | per-agent 决策链、工具调用、消息流、Goal-Plan-Action 拆解 |

**代表框架对比**：

| 框架 | 采集方式 | 分析粒度 | 特点 |
|------|----------|----------|------|
| MASEval | 旁路 per-agent tracing | 系统级 + 单 Agent 级 | 跨框架、解耦设计 |
| MCPEval | MCP 标准协议 | 工具调用级 | 协议化、自动化生成 |
| Agent GPA | 事后 trace 回溯分析 | Goal→Plan→Action 三层 | 无参考 LLM-as-Judge |
| EnvTrustBench | 脚手架内记录 | 证据处理行为 | 关注 grounding 缺陷 |

### 9.6 模式选择指南

| 你的 Agent 类型 | 推荐模式 | 理由 |
|----------------|----------|------|
| 代码生成/开发 Agent | **A 容器沙箱** | 需要真实执行环境验证代码正确性 |
| Web 自动化/RPA Agent | **B 模拟环境** | 需要真实交互验证操作效果 |
| 内容生成 Agent（文本/PPT） | **C 事后评估** | 产出是文件，过程指标需求低 |
| 多 Agent 协作系统 | **D 协议追踪** | 需要系统级行为分析 |
| 需要过程指标的生成类 Agent | **D 协议追踪** 或 **C + 执行 Harness** | 在不控制执行的前提下采集 trace |
