# Agent 能力评估体系指南

> 本文档系统梳理 Agent（智能体）能力评估的行业实践，涵盖评估维度、指标体系、具体评估方法及方法选型建议。

---

## 一、常见的 Agent 能力评估维度

不同场景下的 Agent 评估维度差异很大，但可以抽象出以下通用框架：

### 1.1 通用能力维度（几乎所有 Agent 都需要）

| 维度 | 说明 | 典型场景 |
|------|------|----------|
| **任务完成度** | Agent 是否最终完成了用户指定的目标 | 所有场景 |
| **规划能力 (Planning)** | 是否能将复杂任务分解为合理的子任务序列 | 旅行规划、项目管理 |
| **工具使用 (Tool Use)** | 是否能正确选择、调用、组合外部工具/API | 数据分析、代码生成 |
| **推理能力 (Reasoning)** | 多步逻辑推理、因果推断、数学计算 | 科学研究、金融分析 |
| **记忆能力 (Memory)** | 长上下文记忆、跨轮信息保持、知识更新 | 客服、个人助理 |
| **指令遵循 (Instruction Following)** | 是否严格遵守用户约束和格式要求 | 所有场景 |
| **自我纠错 (Self-correction)** | 发现错误后能否调整策略并重新尝试 | 代码调试、实验设计 |
| **安全性 (Safety)** | 是否产生有害输出、是否越权操作 | 所有场景 |
| **效率 (Efficiency)** | 完成任务所需的步骤数、时间、资源消耗 | 所有场景 |

### 1.2 领域特定维度

| 领域 | 特定维度 |
|------|----------|
| **代码 Agent** | 代码可执行性、测试通过率、代码风格、安全性漏洞 |
| **客服 Agent** | 问题解决率、用户满意度、对话轮数、情绪安抚能力 |
| **研究 Agent** | 信息准确性、引用完整性、报告结构、结论可靠性 |
| **旅行规划 Agent** | 行程可行性、约束满足度、用户偏好匹配、预算合理性 |
| **数据库 Agent** | SQL 正确性、查询效率、Schema 理解准确度 |
| **Web Agent** | 页面操作成功率、信息提取准确率、导航效率 |

## 二、Agent 的评估指标体系

评估指标通常分为 **结果指标**、**过程指标**、**成本指标** 三个层次：

### 2.1 结果指标（Outcome Metrics）

| 指标 | 定义 | 适用场景 |
|------|------|----------|
| **Success Rate** | 任务成功完成的比例 | 通用 |
| **Task Completion Rate** | 子任务完成率（部分完成也算） | 复杂多步任务 |
| **Accuracy** | 输出结果与标准答案的一致性 | 问答、分类 |
| **Constraint Satisfaction Rate** | 约束条件的满足比例 | 规划类任务（如 TripScore 的 DR/CPR） |
| **Hallucination Rate** | 产生幻觉（编造信息）的比例 | 信息检索、知识问答 |
| **User Satisfaction Score** | 用户主观满意度打分 | 客服、助手类 |

### 2.2 过程指标（Process Metrics）

| 指标 | 定义 | 说明 |
|------|------|------|
| **Steps / Turns** | 完成任务所需的步数/轮数 | 步数过多可能意味着效率低或陷入循环 |
| **Tool Calls** | 工具/API 调用次数 | 反映工具使用效率 |
| **Rework Rate** | 重试/修正次数 | 反映自我纠错能力和初始准确性 |
| **Dead-end Rate** | 走入死胡同的比例 | Agent 是否频繁无法推进任务 |
| **Trajectory Quality** | 行动轨迹的合理性 | 中间步骤是否最优或合理 |

### 2.3 成本指标（Cost Metrics）

| 指标 | 定义 |
|------|------|
| **Latency** | 单次响应时间 / 总任务完成时间 |
| **Token Usage** | 输入+输出的 Token 总数 |
| **API Cost** | 实际调用成本（美元） |
| **Compute Cost** | 计算资源消耗 |

## 三、如何进行指标评估？具体方法有哪些？

Agent 评估方法可以按 **"评估主体"** 和 **"评估粒度"** 两个维度分类：

### 3.1 按评估主体分类

```
┌─────────────────────────────────────────────────────────┐
│                    Agent 评估方法体系                      │
├─────────────────┬─────────────────┬─────────────────────┤
│   自动评估       │   人工评估       │    混合评估          │
│  (Automatic)    │   (Human)       │   (Hybrid)          │
├─────────────────┼─────────────────┼─────────────────────┤
│ • Rule-based    │ • 专家评估       │ • TripScore 模式    │
│ • LLM-as-judge  │ • 众包评估       │ • 人-机协同评估      │
│ • 执行验证       │ • A/B 测试       │ • 主动学习评估       │
│ • 仿真评估       │ • 用户反馈       │ • 对抗性评估        │
│ • 基准测试       │ • 人工标注       │                     │
└─────────────────┴─────────────────┴─────────────────────┘
```

### 3.2 按评估粒度分类

| 粒度 | 方法 | 说明 |
|------|------|------|
| **端到端评估** | 给 Agent 完整任务，评估最终结果 | 最接近真实场景，但难以定位问题 |
| **模块级评估** | 单独评估规划、工具调用、记忆等模块 | 便于定位薄弱环节 |
| **轨迹评估** | 评估 Agent 的每一步决策和行动序列 | 能发现过程中的错误 |
| **单步评估** | 评估单次 LLM 调用的输出质量 | 最细粒度，用于调试 |

### 3.3 具体评估方法详解

#### 方法 1：Rule-based Evaluation（规则评估）

- **原理**：预定义明确的判定规则，自动检查 Agent 输出
- **适用**：结构化输出、约束满足类任务
- **示例**：TripScore 的 Format/Commonsense Constraint
- **优缺点**：
  - ✅ 稳定、低成本、可解释
  - ❌ 难以覆盖语义层面的质量

#### 方法 2：LLM-as-judge（模型评判）

- **原理**：使用更强的 LLM（如 GPT-4）作为评判者
- **变体**：
  - **Point-wise**：独立打分（0-100 或 0-1）
  - **Pair-wise**：两两比较选优
  - **Reference-based**：与参考答案对比评分
- **优缺点**：
  - ✅ 灵活、覆盖语义维度
  - ❌ 不稳定、有偏见、成本高

#### 方法 3：Execution-based Evaluation（执行验证）

- **原理**：实际执行 Agent 的输出，验证运行结果
- **适用**：代码生成、SQL 查询、API 调用
- **示例**：
  - 代码 Agent：执行生成的代码，检查测试用例通过率（如 SWE-bench）
  - SQL Agent：在数据库中执行 SQL，对比查询结果
  - 旅行 Agent：验证航班/酒店是否真实存在（TripScore 的信息验证）
- **优缺点**：
  - ✅ 最客观、无幻觉
  - ❌ 安全风险、环境依赖、难以覆盖所有场景

#### 方法 4：Simulation-based Evaluation（仿真评估）

- **原理**：在虚拟/仿真环境中运行 Agent
- **示例**：
  - **WebArena / WebShop**：在模拟网页环境中测试 Web Agent
  - **Minecraft / 游戏环境**：测试具身智能 Agent
  - **browser-use**：在 headless 浏览器中自动化测试
- **优缺点**：
  - ✅ 安全、可重复、可扩展
  - ❌ 仿真与现实有差距

#### 方法 5：Human Evaluation（人工评估）

- **原理**：人类专家或普通用户对 Agent 输出进行评判
- **形式**：
  - **绝对评分**：按维度打分（1-5 分）
  - **相对评分**：并排比较两个输出（Side-by-side）
  - **Elo 评分**：通过多轮对战生成排名分
- **优缺点**：
  - ✅ 最可靠、捕获细微差别
  - ❌ 昂贵、慢、难以扩展

#### 方法 6：Benchmark-based Evaluation（基准测试）

- **原理**：在公开的标准数据集上进行评估
- **知名基准**：

| 基准 | 评估对象 | 评估方式 |
|------|----------|----------|
| **SWE-bench** | 代码 Agent | 执行验证（单元测试） |
| **WebArena** | Web Agent | 仿真环境 + 规则检查 |
| **GAIA** | 通用 Agent | 精确匹配 / 人工验证 |
| **TravelPlanner** | 旅行规划 Agent | 规则检查（通过率） |
| **TripScore** | 旅行规划 Agent | Rule + LLM 混合 |
| **MLAgentBench** | ML 实验 Agent | 执行验证（模型性能） |
| **OSWorld** | 操作系统 Agent | 仿真环境截图对比 |

#### 方法 7：Reference-based Evaluation（参考对比）

- **原理**：将 Agent 输出与标准答案（Ground Truth）对比
- **指标**：
  - **Exact Match**：完全匹配
  - **F1 Score**：部分匹配（适用于文本）
  - **BLEU / ROUGE**：文本相似度（NLP 领域常用）
  - **BERTScore**：基于语义嵌入的相似度
  - **G-Eval**：基于 LLM 的参考对比评分
- **适用**：问答、摘要、翻译等有标准答案的任务

#### 方法 8：Trajectory Evaluation（轨迹评估）

- **原理**：不仅评估最终结果，还评估 Agent 的决策过程
- **方法**：
  - 检查行动序列是否最优
  - 对比专家轨迹（Expert Trajectory）
  - 计算轨迹与最优路径的偏离度
- **适用**：强化学习 Agent、多步决策任务

#### 方法 9：Multi-Agent Evaluation（多智能体评估）

- **原理**：多个 Agent 互相评估或对抗
- **变体**：
  - **辩论式评估**：两个 Agent 辩论，由第三个 Agent 裁决
  - **对抗式评估**：红队 Agent 尝试让被测 Agent 出错
  - **Peer Review**：多个 Agent 互相评审输出

#### 方法 10：Online / A-B Testing（在线评估）

- **原理**：将 Agent 部署到真实环境，收集真实用户反馈
- **指标**：转化率、留存率、用户满意度、人工接管率
- **优缺点**：
  - ✅ 最真实反映效果
  - ❌ 风险高、反馈慢

## 四、除 Rule-based 和 LLM-as-judge 之外的方法汇总

| 方法类别 | 代表方法 | 核心思想 | 适用场景 |
|----------|----------|----------|----------|
| **执行验证** | SWE-bench 模式 | 实际运行输出，看结果 | 代码、SQL、API |
| **仿真环境** | WebArena, OSWorld | 在虚拟环境中运行 | Web、OS、游戏 |
| **人工评估** | 专家打分、众包 | 人类主观判断 | 创意、开放域任务 |
| **参考对比** | BERTScore, G-Eval | 与标准答案对比 | 有标准答案的任务 |
| **轨迹评估** | Expert Trajectory Match | 评估决策过程 | 多步推理任务 |
| **多智能体** | 辩论、红队测试 | Agent 互评互测 | 安全性、可靠性 |
| **在线评估** | A/B Test | 真实用户反馈 | 生产环境 |
| **Reward Model** | 训练专用打分模型 | 学习人类偏好 | RLHF、RL 训练 |
| **对抗性评估** | Adversarial Testing | 构造困难样本测试 | 安全性、鲁棒性 |

## 五、如何选择评估方法？

选择评估方法时需要考虑以下因素：

```
                    高
                    │
    LLM-as-judge   │   Human Evaluation
    (语义质量)      │   (主观体验)
                    │
  评  估  维  度  丰  富  度
                    │
    Rule-based     │   Execution-based
    (结构正确性)    │   (结果正确性)
                    │
                    └────────────────→ 高
                      可扩展性/自动化程度
```

| 场景 | 推荐方法 |
|------|----------|
| 结构化输出 + 明确约束 | Rule-based |
| 开放域生成 + 语义质量 | LLM-as-judge / Human |
| 代码/SQL/可执行输出 | Execution-based |
| 多步交互 + 复杂环境 | Simulation-based |
| 生产环境效果验证 | Online A/B Test |
| 训练过程需要稳定奖励 | Reward Model + Rule Hybrid |
| 最终质量把关 | Human Evaluation |

## 六、行业趋势与最佳实践

1. **混合评估是主流**：单一方法都有明显缺陷，TripScore（Rule + LLM）、SWE-bench（执行 + 规则）都是混合模式的代表
2. **从结果评估转向过程评估**：不仅看 Agent 最终答对了没有，还要看它怎么思考的（如 OpenAI 的 o1 强调的思维链）
3. **成本感知评估兴起**：不仅评估 Agent 做得好不好，还评估它花了多少钱、用了多少 Token
4. **动态/对抗评估**：构造越来越难的测试集，而非静态 benchmark
5. **可解释性要求**：评估结果需要能告诉开发者 Agent 哪里做错了、怎么改
