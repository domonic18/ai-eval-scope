# 课件生成Agent评估体系

> 本文档为"课件自动生成Agent"设计了一套完整的评估体系，覆盖功能性、效果性、安全性、性能四大维度，借鉴 TripScore 约束检查架构、OpenCompass 级联评估方法及行业通用评估框架，形成可落地实施的评估方案。

> **⚠️ 适配说明（2026-06-08）**：当前阶段评估目标已从 PPTX 课件调整为 **Markdown/HTML 文档集**（多文档组合），暂不支持 PPTX。本文档中涉及的"幻灯片""页数""PPTX"等描述为原始需求设计，具体评估器实现已按 Markdown/HTML 适配，详见 [04 评估引擎设计](../arch/04评估引擎设计.md)。

## 一、背景与目标

### 1.1 背景

课件生成Agent面向教师用户，根据教学需求（学科、年级、知识点、课时等）自动生成结构化的教学课件（PPT/Slides）。其输出物具有特殊性：

- **多模态输出**：包含文本、图片、图表、版式布局等
- **教育属性强**：内容必须准确，符合教学大纲和课程标准
- **结构化要求**：遵循"导入→新授→练习→总结"等教学逻辑
- **视觉呈现**：排版、配色、字体等影响课堂使用效果
- **受众差异化**：需适配不同年级、学科、教学风格

### 1.2 评估目标

| 目标 | 说明 |
|------|------|
| **质量可控** | 建立可量化的课件质量评估标准 |
| **问题定位** | 能定位Agent薄弱环节（内容？格式？教学逻辑？） |
| **迭代指导** | 为Agent优化提供明确方向和数据支撑 |
| **上线把关** | 形成上线前的质量验收机制 |

### 1.3 设计原则

1. **借鉴 TripScore "约束检查 → 分数聚合" 架构**：将课件质量拆分为可逐项判定的约束条件
2. **级联评估**：先过硬性门控（知识准确、格式合规），再评软质量（教学设计、视觉美观）
3. **结果+过程+成本三层覆盖**：不仅看课件好不好，还要看生成效率和资源消耗
4. **自动化优先**：核心指标尽量自动评估，降低人工成本

## 二、课件生成Agent的核心能力模型

在通用Agent能力模型基础上，课件生成Agent需要额外关注以下特有能力：

```
┌─────────────────────────────────────────────────────────────┐
│                  课件生成Agent 能力金字塔                      │
├─────────────────────────────────────────────────────────────┤
│  第4层: 教学专业层                                            │
│         知识点准确 · 教学目标对齐 · 难度适配 ·  pedagogical设计 │
├─────────────────────────────────────────────────────────────┤
│  第3层: 内容生成层                                            │
│         文本质量 · 图片/图表生成 · 多媒体元素整合 · 版权合规     │
├─────────────────────────────────────────────────────────────┤
│  第2层: 结构组织层                                            │
│         课件结构完整 · 教学逻辑清晰 · 页面布局合理 · 导航清晰     │
├─────────────────────────────────────────────────────────────┤
│  第1层: 基础能力层                                            │
│         指令理解 · 约束遵循 · 信息检索 · 工具调用 · 安全合规     │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 四层能力详述

| 层级 | 核心能力 | 评估重点 |
|------|----------|----------|
| **L1 基础能力** | 理解教师需求、遵循格式约束、调用外部工具 | 指令遵循率、工具调用准确率 |
| **L2 结构组织** | 生成完整课件结构、合理页面布局、清晰导航 | 结构完整性、格式合规性 |
| **L3 内容生成** | 生成准确的教学文本、合适的配图/图表 | 内容准确性、多媒体质量 |
| **L4 教学专业** | 知识点准确、教学目标达成、难度适配学生 | 教学适用性、教育专业性 |

## 三、评估体系总体架构

### 3.1 三层指标架构

```
用户请求（学科/年级/知识点/偏好）
           ↓
    课件生成Agent 执行
           ↓
    ┌──────────────────────────────────────┐
    │           评估指标体系                  │
    ├──────────────────────────────────────┤
    │  结果指标（Outcome）                  │
    │  · 课件质量 · 任务完成 · 内容准确     │
    ├──────────────────────────────────────┤
    │  过程指标（Process）                  │
    │  · 生成步数 · 重试次数 · 死胡同率     │
    ├──────────────────────────────────────┤
    │  成本指标（Cost）                     │
    │  · 响应时间 · Token消耗 · API费用     │
    └──────────────────────────────────────┘
           ↓
    约束检查（4大类约束逐项判定）
           ↓
    综合质量分数（Reward Score）
```

### 3.2 四大评估维度

| 维度 | 核心关注点 | 对应指标层级 |
|------|-----------|-------------|
| **功能性** | Agent是否完成了课件生成任务 | 结果指标 |
| **效果性** | 生成的课件质量如何 | 结果指标 |
| **安全性** | 是否产生有害/错误/偏见内容 | 结果指标 |
| **性能** | 生成效率和资源消耗 | 过程指标 + 成本指标 |

## 四、评估维度与指标体系

### 4.1 功能性指标

功能性评估Agent是否成功完成了教师指定的课件生成任务。

#### 4.1.1 Success Rate（任务成功率）

**定义**：Agent成功生成可用课件的任务比例。

**计算方法**：

```
Success Rate = 成功生成课件的任务数 / 总任务数 × 100%
```

**判定标准**（需同时满足）：
- 生成了完整的课件文件（PPT/PDF等）
- 课件可正常打开浏览
- 未发生系统错误或超时

#### 4.1.2 Task Completion Rate（子任务完成率）

**定义**：课件生成过程中各子任务的完成情况。课件生成通常包含多个子任务（大纲生成→内容撰写→配图生成→排版美化）。

**计算方法**：

```
Task Completion Rate = 完成的子任务数 / 总子任务数 × 100%
```

**典型子任务拆分**：

| 子任务 | 说明 |
|--------|------|
| 大纲生成 | 根据需求生成课件结构大纲 |
| 页面生成 | 为每个大纲节点生成页面内容 |
| 配图生成/检索 | 为页面匹配合适的图片 |
| 图表生成 | 生成数据可视化图表 |
| 排版美化 | 调整字体、配色、布局 |
| 导出渲染 | 输出为最终课件文件 |

---

### 4.2 效果性指标

效果性评估生成课件的质量，是评估体系的核心。

#### 4.2.1 Constraint Satisfaction Rate（约束满足率）

**定义**：教师输入的各类约束条件被满足的比例。

**计算方法**：

```
Constraint Satisfaction Rate = 满足的约束项数 / 总约束项数 × 100%
```

**课件场景的典型约束分类**：

| 约束类别 | 示例 |
|----------|------|
| **格式约束** | 页数限制、幻灯片比例（16:9/4:3）、字体要求 |
| **内容约束** | 必须包含的知识点、指定的教学案例 |
| **风格约束** | 配色主题、动画效果要求、简洁/详细风格 |
| **结构约束** | 必须包含导入/练习/总结环节 |

#### 4.2.2 Accuracy（内容准确率）

**定义**：课件中教学内容与标准知识体系的一致性。

**计算方法**（参考 OpenCompass BERTScore + Exact Match 混合）：

```
Accuracy = w1 × ExactMatch + w2 × BERTScore
```

其中：
- **ExactMatch**：知识点名称、公式、定义等精确匹配率
- **BERTScore**：基于语义嵌入的内容相似度（适合概念解释类文本）
- 权重建议：$w_1 = 0.6, w_2 = 0.4$（精确知识优先）

**评估方式**：
1. 提取课件中的知识点陈述
2. 与标准教材/课标中的对应知识点对比
3. 逐条判定是否准确

#### 4.2.3 Knowledge Coverage Rate（知识点覆盖率）

**定义**：课件覆盖的教学知识点占目标知识点集合的比例。

**计算方法**：

```
Knowledge Coverage Rate = |课件覆盖的知识点| / |目标知识点集合| × 100%
```

**适用场景**：教师指定了明确的教学目标和知识点清单。

#### 4.2.4 Hallucination Rate（幻觉率）

**定义**：课件中编造事实、虚构知识点、错误引用的比例。

**计算方法**（参考 TripScore 信息验证机制）：

```
Hallucination Rate = 存在幻觉的课件数 / 总评估课件数 × 100%
```

**幻觉类型**（课件场景特化）：

| 类型 | 示例 |
|------|------|
| **事实性幻觉** | 历史年代错误、科学公式错误 |
| **概念性幻觉** | 对专业术语的错误解释 |
| **引用幻觉** | 虚构不存在的参考文献 |
| **图表数据幻觉** | 图表中的数据无来源或错误 |

**检测方法**：
1. **Rule-based**：公式、年代、数值等可精确验证的内容
2. **LLM-as-judge**：概念解释、事实陈述等需语义理解的内容
3. **人工抽检**：复杂知识的人工复核

#### 4.2.5 Pedagogical Quality Score（教学适用性评分）

**定义**：课件在教学设计层面的专业质量。

**评估维度**（5个子项）：

| 子项 | 权重 | 评估内容 |
|------|------|----------|
| **教学目标对齐** | 25% | 课件内容是否与声明的教学目标一致 |
| **难度适配性** | 20% | 内容难度是否匹配目标年级/学生水平 |
| **教学逻辑性** | 20% | 内容组织是否符合认知规律（由浅入深） |
| **互动性设计** | 15% | 是否包含提问、讨论、练习等互动环节 |
| **评价与反馈** | 20% | 是否包含学习效果检测和反馈机制 |

**计算方法**：

```
Pedagogical Quality Score = Σ(子项得分 × 子项权重)
```

**评估方式**：LLM-as-judge（使用教育专家Prompt）+ 人工抽检校准。

#### 4.2.6 Visual Quality Score（视觉质量评分）

**定义**：课件的视觉呈现质量。

**评估维度**（4个子项）：

| 子项 | 权重 | 评估内容 |
|------|------|----------|
| **排版规范性** | 30% | 字体统一、对齐正确、间距合理 |
| **视觉美观度** | 25% | 配色协调、图文搭配得当 |
| **信息层级** | 25% | 标题/正文/注释层级清晰 |
| **可读性** | 20% | 字号适中、对比度足够、无视觉干扰 |

**评估方式**：
- **Rule-based**：字体数量、字号范围、对齐方式等可量化指标
- **LLM-as-judge**：视觉美观度、配色协调性等主观维度
- **可选**：布局分析工具（如检测文字重叠、留白比例）

#### 4.2.7 Multimedia Quality Score（多媒体质量评分）

**定义**：课件中图片、图表、视频等多媒体元素的质量。

**评估维度**：

| 子项 | 评估内容 |
|------|----------|
| **图片相关性** | 配图是否与内容语义匹配 |
| **图片清晰度** | 分辨率是否满足投影需求 |
| **图表准确性** | 图表数据是否正确、类型是否恰当 |
| **版权合规性** | 图片是否有版权风险 |

---

### 4.3 安全性指标

#### 4.3.1 Harmful Content Rate（有害内容率）

**定义**：课件中包含有害内容（暴力、色情、歧视、政治敏感等）的比例。

**计算方法**：

```
Harmful Content Rate = 含有害内容的课件数 / 总评估课件数 × 100%
```

**检测方法**：
1. **关键词过滤**：敏感词黑名单检测
2. **LLM-as-judge**：语义层面的有害内容识别
3. **图片安全检测**：配图是否包含不适宜内容

#### 4.3.2 Bias Detection Rate（偏见检出率）

**定义**：课件中存在性别、种族、地域、文化等偏见内容的比例。

**课件场景的典型偏见类型**：

| 类型 | 示例 |
|------|------|
| **性别偏见** | 科学家示例全是男性 |
| **文化偏见** | 以单一文化视角描述历史事件 |
| **地域偏见** | 地域刻板印象描述 |
| **学科偏见** | 某些学科被贬低的表述 |

---

### 4.4 性能指标

#### 4.4.1 响应速度指标

| 指标 | 定义 | 计算方法 |
|------|------|----------|
| **Latency** | 单次响应时间 | 从发送请求到收到首字节的时间（秒） |
| **Total Completion Time** | 总任务完成时间 | 从发送请求到课件生成完成的总时间（秒） |
| **Time-to-First-Slide** | 首页生成时间 | 从请求到第一张幻灯片生成的时间（秒） |

#### 4.4.2 成本消耗指标

| 指标 | 定义 | 计算方法 |
|------|------|----------|
| **Token Usage** | 输入+输出的Token总数 | Prompt Token + Completion Token |
| **API Cost** | 实际调用成本 | 按模型定价计算的总费用（美元/人民币） |
| **Image Generation Cost** | 图片生成成本 | 若使用DALL-E/Midjourney等，单独计算 |

#### 4.4.3 过程效率指标

| 指标 | 定义 | 计算方法 | 意义 |
|------|------|----------|------|
| **Steps / Turns** | 完成任务所需的步数/轮数 | Agent执行过程中的步骤计数 | 步数过多可能意味着规划低效或陷入循环 |
| **Rework Rate** | 重试/修正次数 | 某一步骤失败后的重试次数 | 反映初始准确性 |
| **Tool Calls** | 工具/API调用次数 | 外部工具（图片生成、搜索等）调用次数 | 工具使用效率 |
| **Dead-end Rate** | 走入死胡同的比例 | 无法继续推进的任务比例 | 是否频繁卡住 |

## 五、课件专用约束检查框架

借鉴 TripScore 的"约束检查 → 分数聚合"架构，将课件质量评估拆分为 **4大类、16个细项** 约束条件。

### 5.1 四类约束总览

| 约束类别 | 性质 | 评估方式 | 失败惩罚 |
|----------|------|----------|----------|
| **格式约束** | 硬性门控 | Rule-based | 高（-3） |
| **常识约束** | 硬性门控 | Rule + 事实验证 | 高（-1） |
| **软约束** | 质量提升 | Rule + LLM | 低（归一化得分） |
| **偏好约束** | 个性化 | LLM-as-judge | 低（归一化得分） |

### 5.2 格式约束（Format Constraint）— 4项，全部 Rule-based

确保课件的格式完整性和可用性。

| 子项 | 评估内容 | 检查方式 |
|------|----------|----------|
| **Response Format** | 输出是否为有效的课件格式（PPTX/PDF） | 文件头/扩展名检查 |
| **Slide Count** | 页数是否在要求的范围内 | 计数检查 |
| **Aspect Ratio** | 幻灯片比例是否符合要求（16:9/4:3） | 尺寸计算 |
| **Font Availability** | 使用的字体是否为通用字体 | 字体白名单检查 |

### 5.3 常识约束（Commonsense Constraint）— 5项，Rule + 事实验证

检验课件内容的逻辑合理性和知识准确性。

| 子项 | 评估内容 | 评估方式 |
|------|----------|----------|
| **Information Accuracy** | 知识点、公式、年代、数据是否准确 | 与权威知识库对比 |
| **Chronological Order** | 历史事件、发展流程是否按正确时序 | 时序规则检查 |
| **Logical Consistency** | 内容前后逻辑是否自洽 | LLM一致性检查 |
| **Math Formula Correctness** | 数学公式是否正确书写和计算 | 公式解析验证 |
| **Unit Consistency** | 物理量单位是否统一、正确 | 单位规则检查 |

### 5.4 软约束（Soft Constraint）— 4项，Rule + LLM 混合

评估课件的教学质量和用户体验。

| 子项 | 评估方式 | 评估内容 |
|------|----------|----------|
| **Content Density** | Rule | 每页信息量适中，不过于拥挤或稀疏 |
| **Visual Consistency** | Rule | 配色、字体、风格在全课件中保持一致 |
| **Teaching Logic** | LLM | 内容组织符合认知规律（导入→新授→练习→总结） |
| **Content Diversity** | LLM | 内容呈现方式多样化（文字、图片、图表、案例） |

### 5.5 偏好约束（Preference Constraint）— 3项，LLM-as-judge

评估课件是否符合教师的个性化需求。

| 子项 | 评估方式 | 评估内容 |
|------|----------|----------|
| **Style Preference** | LLM | 是否符合要求的风格（简洁/详细/活泼/正式） |
| **Depth Preference** | LLM | 内容深度是否符合要求（概览/深入/拓展） |
| **User Request Fulfillment** | LLM | 是否满足教师在查询中的特定要求 |

## 六、综合评分公式

### 6.1 奖励分数计算

借鉴 TripScore 的加权奖励公式，课件综合质量分计算如下：

```
Reward = S_format + S_common + w3 × S_soft + w4 × S_pref
```

| 符号 | 含义 | 计算方式 |
|------|------|----------|
| $S_{format}$ | 格式约束得分 | 通过=+1，失败=-3（硬性门控） |
| $S_{common}$ | 常识约束得分 | 通过=+1，失败=0 |
| $S_{soft}$ | 软约束平均得分 | 各子项归一化到[0,1]后加权平均 |
| $S_{pref}$ | 偏好约束平均得分 | 各子项归一化到[0,1]后加权平均 |
| $w_3$ | 软约束乘数 | 默认1.0 |
| $w_4$ | 偏好乘数 | 默认1.0（可调整以反映个性化重要性） |

### 6.2 门控机制

采用严格的层级判定：

1. **第一层 — 格式约束**：若格式检查失败，$S_{format} = -3$，评估立即终止
2. **第二层 — 常识约束**：若常识检查失败（知识错误），$S_{common} = 0$
3. **第三层 — 软约束与偏好约束**：格式和常识均通过后，软约束和偏好约束累加贡献最终奖励

### 6.3 核心评估指标定义

| 指标 | 全称 | 含义 | 计算方式 |
|------|------|------|----------|
| **DR** | Delivery Rate | 交付率：满足所有格式约束的课件比例 | 格式约束全通过的课件数 / 总数 |
| **CPR** | Commonsense Pass Rate | 常识约束通过率：满足所有常识约束的课件比例 | 常识约束全通过的课件数 / 总数 |
| **Reward** | Reward Score | 综合奖励分数 | 上述加权公式 |
| **CondR** | Conditional Reward | 条件奖励：仅在通过DR和CPR的课件上计算的Reward | 通过门控的课件的Reward均值 |
| **Time** | Total Completion Time | 平均总完成时间（秒） | 所有任务完成时间的平均值 |

## 七、评估方法与工具选型

### 7.1 方法矩阵

| 评估维度 | 推荐方法 | 工具/实现 | 优先级 |
|----------|----------|-----------|--------|
| 格式合规性 | Rule-based | Python-pptx 文件解析 | P0 |
| 知识准确性 | 事实验证 + LLM | 知识库API + GPT-4 | P0 |
| 教学适用性 | LLM-as-judge | GPT-4o + 教育专家Prompt | P0 |
| 视觉质量 | Rule + LLM | 布局分析 + 视觉评估Prompt | P1 |
| 幻觉检测 | Rule + LLM | math_verify + GPT-4 | P0 |
| 安全检测 | Rule + LLM | 敏感词过滤 + 内容审核API | P0 |
| 过程效率 | 自动统计 | Agent执行日志 | P1 |
| 成本消耗 | 自动统计 | Token计数 + API账单 | P1 |

### 7.2 级联评估流程（推荐）

借鉴 OpenCompass CascadeEvaluator 的设计：

```
输入课件
  ↓
[第一层] Rule-based 快速筛选
  · 格式检查（文件有效性、页数、尺寸）
  · 关键词安全过滤
  · 若失败 → 直接判定为不合格，记录错误类型
  ↓ 通过
[第二层] 事实验证
  · 知识点与权威知识库对比
  · 公式/数据精确验证
  · 若失败 → 标记为知识错误，终止评分
  ↓ 通过
[第三层] LLM-as-judge 质量评估
  · 教学适用性评分
  · 视觉质量评分
  · 偏好满足度评分
  ↓
输出综合 Reward Score
```

**优势**：
- 大部分不合格课件在低成本阶段被快速筛除
- 仅高质量课件进入高成本的 LLM 评估阶段
- 可解释性强：每个失败都有明确的约束项对应

### 8.2 数据集构建建议

| 数据集类型 | 数量建议 | 构建方式 |
|------------|----------|----------|
| **格式测试集** | 50+ | 覆盖不同格式约束（页数、比例、风格） |
| **学科覆盖集** | 200+ | 覆盖语文/数学/英语/物理/化学/历史等主要学科 |
| **难度梯度集** | 100+ | 同知识点不同年级版本（小学/初中/高中） |
| **边缘案例集** | 30+ | 极端需求（1页极简课件、50页详细课件等） |
| **对抗测试集** | 20+ | 包含敏感/偏见/错误知识点的故意测试 |

### 8.3 质量控制

采用三层过滤确保评估质量：

1. **第一层 — 自动过滤**：排除Agent执行崩溃的样本
2. **第二层 — 规则过滤**：排除格式明显不合规的样本（单独记录）
3. **第三层 — 人工抽检**：对LLM评估结果进行10%抽检，校准评分标准

## 九、实施建议与迭代路线

### 9.1 实施路线图

| 阶段 | 周期 | 目标 | 关键产出 |
|------|------|------|----------|
| **Phase 1: MVP** | 2周 | 建立基础评估能力 | 格式约束检查 + 安全检测 + 基础指标统计 |
| **Phase 2: 核心** | 4周 | 覆盖质量评估 | 加入知识准确性 + 教学适用性 + Reward Score |
| **Phase 3: 完善** | 4周 | 提升评估深度 | 视觉质量 + 多媒体质量 + 偏见检测 |
| **Phase 4: 运营** | 持续 | 常态化评估 | 自动评估Pipeline + 评估报告 + 迭代优化 |

### 9.2 关键决策建议

| 决策点 | 建议 | 理由 |
|--------|------|------|
| **评估频率** | 每次迭代/每周 | 课件Agent迭代快，需持续监控 |
| **基准模型** | 保留1-2个稳定版本作为对比基线 | 便于对比迭代效果 |
| **人工介入** | 10%抽检 + 争议案例仲裁 | 平衡成本与可靠性 |
| **阈值设定** | DR ≥ 95%, CPR ≥ 90%, Reward ≥ 0.7 | 教育场景容错率低 |

### 9.3 常见问题与对策

| 问题 | 对策 |
|------|------|
| LLM评估不稳定 | 使用温度≈0，固定seed，多次采样取平均 |
| 知识库更新滞后 | 建立定期更新机制，对争议知识点人工仲裁 |
| 视觉评估主观性强 | 建立视觉评估标准样例库，统一评分尺度 |
| 评估成本高 | 级联评估优先，大部分样本在Rule阶段完成 |

#### 教学适用性评估 Prompt（LLM-as-judge）

```
你是一位资深教育专家。请对以下课件进行教学适用性评估，从5个维度打分（0-10分）：

1. 教学目标对齐度：课件内容是否与声明的教学目标一致
2. 难度适配性：内容难度是否匹配目标年级学生
3. 教学逻辑性：内容组织是否符合认知规律
4. 互动性设计：是否包含提问、练习等互动环节
5. 评价与反馈：是否包含学习效果检测

请输出每个维度的得分和简要理由，格式为JSON。
```

#### 视觉质量评估 Prompt（LLM-as-judge）

```
你是一位资深视觉设计师。请对以下课件的视觉呈现进行质量评估，从4个维度打分（0-10分）：

1. 排版规范性：字体统一、对齐正确、间距合理
2. 视觉美观度：配色协调、图文搭配得当
3. 信息层级：标题/正文/注释层级清晰
4. 可读性：字号适中、对比度足够

请输出每个维度的得分和简要理由，格式为JSON。
```
