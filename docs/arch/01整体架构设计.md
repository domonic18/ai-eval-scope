## 一、设计目标与约束

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **执行-评估解耦** | 执行引擎与评估引擎独立运行、独立演进，通过标准化数据包交互 |
| **通用可扩展** | 以课件生成为切入点，架构上支持代码生成、RAG、对话等多类 Agent 评估 |
| **Agent 优先** | CLI / Python SDK 为首要接口，便于 Claude Code 等工具调用 |
| **规则即配置** | 评估规则以声明式 YAML 描述，支持 Agent 自动生成与人类校对 |
| **模型无关** | LLM-as-judge 通过 Provider 抽象层接入，支持多厂商多协议 |
| **级联短路** | 低成本检查在前，失败即终止，避免不必要的 LLM 调用 |
| **离线优先** | 前期批量离线评估为主，接口预留流式扩展空间 |

### 1.2 架构约束

- **技术栈**：Python 3.11+
- **部署形态**：前期 CLI + 可选 REST API → 中期 FastAPI 服务 → 远期 SaaS
- **SUT 接入**：以 HTTP API 为主（LangGraph / Hermes），保留 CLI/本地进程扩展点
- **数据存储**：前期文件系统（workspace 目录）→ 远期数据库
- **认证授权**：本阶段不考虑

### 2.3 数据流概览

```
task_set.yaml ──→ 执行引擎 ──→ ExecutionPackage ──→ 编排调度层 ──→ 评估引擎 ──→ EvaluationResult
                       ↑                      ↑                      ↑
                  sut_config.yaml        workspace/runs/         rule_set.yaml
```

三种运行模式：

| 模式 | 说明 |
|------|------|
| **pipeline** | 执行 + 评估串行运行，一步到位 |
| **run-only** | 仅执行，生成 ExecutionPackage 到 workspace |
| **eval-only** | 仅评估，从 workspace 读取已有 ExecutionPackage |

## 四、技术选型

| 用途 | 推荐库 | 说明 |
|------|--------|------|
| CLI | `typer` | 现代 Python CLI |

## 六、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 执行-评估交互 | 文件系统数据包（workspace/） | 简单、可审计、便于复用 |
| SUT 默认驱动 | HTTPDriver | 被测 Agent 主要是 HTTP 服务 |
| 规则格式 | YAML + JSON Schema | 人可读、Agent 易解析 |
| LLM 接入 | Provider 抽象层 | 屏蔽协议差异 |
| 级联策略 | fail_fast + continue_all | 低成本筛除 + 完整诊断 |
| 缓存策略 | 输入 Hash + 本地缓存 | 降低 LLM 调用成本 |
| 文档组织 | 总览 + 按架构层次拆分 | 便于维护和扩展 |
