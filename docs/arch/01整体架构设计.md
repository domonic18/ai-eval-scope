## 一、设计目标与约束

### 1.1 设计目标

| 目标 | 说明 |
|------|------|
| **Agent 驱动执行** | 执行引擎由 Claude Agent SDK 端到端驱动，Agent 理解任务、智能调用 SUT、采集结果 |
| **执行-评估解耦** | 执行引擎与评估引擎独立运行、独立演进，通过标准化数据包交互 |
| **评估双模式** | 评估引擎支持 Pipeline（默认，确定性级联）和 Agent（可选，自适应）两种模式 |
| **通用可扩展** | 以课件生成为切入点，架构上支持代码生成、RAG、对话等多类 Agent 评估 |
| **模型无关** | LLM-as-judge 通过 Provider 抽象层接入，支持多厂商多协议；执行 Agent 使用 Claude |
| **离线优先** | 前期批量离线评估为主，接口预留流式扩展空间 |
| **可视化可追溯** | 提供 Web Portal 展示评估趋势与详细报告，支持按项目维度回溯 |

### 1.2 架构约束

- **技术栈**：Python 3.11+
- **Agent SDK**：`claude-agent-sdk`，执行引擎通过 Claude Agent SDK 端到端驱动
- **部署形态**：前期 CLI + Web Portal + REST API → 中期 FastAPI 服务 → 远期 SaaS
- **Web 框架**：React + Express，适配腾讯云函数部署
- **SUT 接入**：SUT 能力通过 MCP Tools 暴露给执行 Agent（HTTP 调用、CLI 调用、文件操作）
- **评估目标**：以 Markdown/HTML 文档集为主要评估对象（支持目录结构输入），暂不支持 PPTX
- **数据存储**：前期文件系统（workspace 目录）→ 远期数据库
- **认证授权**：本阶段不考虑

### 2.3 数据流概览

```
task_set.yaml ──→ ExecutionAgent (claude-agent-sdk) ──→ SUT Tools (MCP) ──→ ExecutionPackage
       │                                                     │
       │        Agent 理解任务描述，智能调用 SUT              │
       │        通过 MCP Tools 实现 HTTP/CLI 调用            │
       │        自动采集结果并写入 ExecutionPackage           │
       └─────────────────────────────────────────────────────┘
                                       │
                                       ▼
               ┌───────────────────────────────────────────────┐
               │           评估引擎（双模式）                    │
               │                                               │
               │  模式一：PipelineEngine（默认，确定性）         │
               │  Stage1 格式门控 → Stage2 常识检查              │
               │  → Stage3 质量评估 → ScoreAggregator           │
               │                                               │
               │  模式二：EvaluationAgent（可选，自适应）        │
               │  读取 Markdown 评测计划 → MCP Tools 调用评估器  │
               └───────────────────────┬───────────────────────┘
                                       │
                                       ▼
                           EvaluationResult + MetricsReport
                                       │
                                       ▼
                           ReportGenerator → workspace/reports/
                           IndexUpdater   → workspace/index/
```

三种运行模式：

| 模式 | 说明 |
|------|------|
| **pipeline** | ExecutionAgent 执行 → 评估引擎评估 → 报告 |
| **run-only** | 仅 ExecutionAgent 执行，生成 ExecutionPackage |
| **eval-only** | 从 workspace 加载 ExecutionPackage → 评估引擎评估 → 报告 |

### 2.4 核心机制

本系统采用**执行 Agent 化 + 评估双模式**架构：

| 机制 | 说明 |
|------|------|
| **执行 Agent 化** | ExecutionAgent (claude-agent-sdk) 端到端驱动执行：理解任务 → 构建请求 → 调用 SUT → 采集结果 |
| **SUT 即 MCP Tools** | SUT 交互能力（HTTP 调用、CLI 调用、文件操作）通过 MCP Tools 暴露给执行 Agent |
| **评估 Pipeline 默认** | PipelineEngine + PipelineStage 执行固定级联流程，确定性高、无额外 token 消耗 |
| **评估 Agent 可选** | EvaluationAgent 读取 Markdown 评测计划，自适应评估，用于复杂场景 |
| **评分确定性** | 评分聚合始终由 ScoreAggregator（确定性代码）执行，无论评估模式 |
| **预算控制** | `max_turns` + `max_budget_usd` 双重控制，防止执行 Agent 成本失控 |

**设计原则**：执行智能化（Agent 理解任务、自适应执行），评估确定性（Pipeline 默认、Agent 可选）。

## 六、关键设计决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| 执行引擎 | ExecutionAgent (claude-agent-sdk) | Agent 理解任务、智能调用 SUT、自适应处理异常 |
| 评估引擎 | PipelineEngine（默认）+ EvaluationAgent（可选） | 评估流程固定，Pipeline 确定性高且无额外 token；Agent 模式用于复杂自适应场景 |
| SUT 交互方式 | SUT Tools (MCP) | HTTP/CLI/文件操作包装为 MCP Tools，Agent 按需调用 |
| 执行-评估交互 | 文件系统数据包（workspace/） | 简单、可审计、便于复用 |
| 级联编排 | PipelineStage 固定级联（默认）+ Markdown 评测计划（Agent 模式） | 默认确定性流程；Agent 模式下更灵活 |
| 规则格式 | YAML + JSON Schema | 人可读、Agent 易解析 |
| LLM 接入 | Provider 抽象层 | 屏蔽协议差异 |
| 文档组织 | 总览 + 按架构层次拆分 | 便于维护和扩展 |
| Web Portal | React + Express | 适配腾讯云函数部署，前后端分离 |
| 评估输入 | 支持目录结构 | DirectoryCollector 遍历目录，manifest 记录层级 |
| 评分确定性 | ScoreAggregator 代码执行 | 评分计算始终保持确定性，不依赖 Agent 推理 |
