# 评估指标与方法

> 本文档完整描述系统的评估指标体系，包括全部 10 个核心评估器（另含 1 个视觉评估器 `vision.quality`，`--enable-vision` 时 opt-in）的指标定义、评估方法（Rule-based / LLM-as-Judge）、检查算法、评分计算方法，以及相关配置文件的维护位置。属于 [04 评估引擎设计](./04评估引擎设计.md) 的详细展开。

---

## 一、总览

### 1.1 评估体系概览

系统采用**三阶段级联**评估架构，包含 **10 个核心评估器**（另含 1 个视觉评估器 `vision.quality`，`--enable-vision` 时 opt-in），分为四个约束层级：

| 阶段 | 评估器数 | 层级 | 失败行为 | 评估方法 |
|------|---------|------|---------|---------|
| **Stage 1: format** | 2 | HARD_GATE | 任一失败 → S_format = 0（不得分），后续阶段全部 SKIP | Rule-based |
| **Stage 2: commonsense** | 3 | HARD_SCORE | 任一失败 → S_common = 0，不中断阶段内其他评估器，不阻塞后续阶段 | Rule + LLM |
| **Stage 3: quality** | 5 | SOFT(2) + PREFERENCE(3) | 归一化 [0,1] 加权平均，不影响门控 | LLM-as-Judge |

**完整评估器列表**：

| # | evaluator_id | 名称 | 阶段 | 层级 | 方法 |
|---|---|---|---|---|---|
| 1 | `format.response_format` | 文件格式检查 | format | HARD_GATE | RULE |
| 2 | `format.html_validity` | HTML 有效性检查 | format | HARD_GATE | RULE |
| 3 | `commonsense.info_accuracy` | 知识准确性检查 | commonsense | HARD_SCORE | FACT_VERIFY |
| 4 | `commonsense.chronological_order` | 时序正确性检查 | commonsense | HARD_SCORE | LLM_JUDGE |
| 5 | `commonsense.logical_consistency` | 逻辑一致性检查 | commonsense | HARD_SCORE | LLM_CONSISTENCY |
| 6 | `soft.teaching_logic` | 教学逻辑 | quality | SOFT | LLM_JUDGE |
| 7 | `soft.content_diversity` | 内容多样性 | quality | SOFT | LLM_JUDGE |
| 8 | `pref.style_preference` | 风格偏好 | quality | PREFERENCE | LLM_JUDGE |
| 9 | `pref.depth_preference` | 深度偏好 | quality | PREFERENCE | LLM_JUDGE |
| 10 | `pref.request_fulfillment` | 需求满足度 | quality | PREFERENCE | LLM_JUDGE |
| 附 | `vision.quality` | 视觉质量 | vision | SOFT | VISION（opt-in） |

> 评估器可经 RuleSet 的 `enabled` 字段按需关闭（见 [04 §四 RuleSet](./04评估引擎设计.md)）；未在 RuleSet 中显式声明的评估器默认启用。

### 1.2 评估方法分类

| 方法标识 | 说明 | 使用的评估器 |
|---------|------|-------------|
| `RULE` | 纯规则匹配（正则、文件属性检查），无需 LLM | format 全部 |
| `FACT_VERIFY` | 事实性验证（规则检查 + 规则误报经 LLM 二次确认 + LLM 语义验证） | info_accuracy |
| `LLM_CONSISTENCY` | 逻辑一致性验证（LLM 模式为主，Rule-based 降级） | logical_consistency |
| `LLM_JUDGE` | 纯 LLM 评估（多次采样取中位数） | quality 阶段全部 5 个、chronological_order |
| `VISION` | 截图 + LLM 视觉评估（多模态） | vision.quality（opt-in） |

### 1.3 公共基础设施

**样本路径解析**：所有评估器通过 `_get_output_dir(sample)` 获取输出目录，支持 Path / dict / 有 `output_dir` 属性的对象。

**文本内容收集**：Commonsense 和 Quality 阶段的评估器使用 `_collect_text_content(output_dir)` 或 `_collect_file_texts(output_dir)` 将文档内容合并。遍历 `*.md, *.html, *.htm` 文件，HTML 文件去除标签后以换行拼接。

## 二、格式约束指标（2 项）— HARD_GATE

> **评估方法**：全部为 RULE，无需 LLM。
> **通过条件**：全部 2 个评估器均 PASS → S_format = +1；任一 FAIL → S_format = 0（不得分，严重性由 DR + fail-fast 体现）。
> **配置文件**：`assets/rules/default_rule_set.yaml`

---

### 2.1 format.response_format — 文件格式检查

**指标定义**：验证所有输出文件的扩展名和内容头是否为有效的 Markdown 或 HTML。

**检查算法**：

```
1. 从 params["allowed_formats"] 获取允许格式列表（默认 ["md", "html"]）
2. 构建扩展名映射：md→[".md",".markdown"], html→[".html",".htm"]
3. rglob("*") 遍历 output_dir 下所有文件（排除 _manifest.json）
4. 对每个文件：
   a. 扩展名匹配 → 进入内容头校验：
      - .md/.markdown：前 500 字符非空即通过
      - .html/.htm：前 500 字符以 "<" 开头 或 包含 "<!doctype" 或 "<html"
   b. 扩展名不匹配 → 标记无效
```

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `allowed_formats` | `["md", "html"]` | 允许的文件格式列表 |

**判定标准**：全部通过 → score=1.0 PASS；任一不通过 → score=0.0 FAIL

**已知局限**：Markdown 内容头检查过于宽松（仅判断非空）；不检查文件编码。

---

### 2.2 format.html_validity — HTML 有效性检查

**指标定义**：验证 HTML 文件标签嵌套是否平衡、结构是否有效。

**检查算法**（基于标准库 `html.parser.HTMLParser` 的 `_TagStackValidator`，从语义层而非字符串层判断）：

```
1. 收集 .html/.htm 文件
2. check_html_only=True 且无 HTML 文件 → 自动 PASS
3. 对每个文件：
   a. 空文件检查：content.strip() 为空 → FAIL
   b. HTML 结构：以 "<" 开头 或 包含 "<!doctype" 或 "<html"
   c. 标签栈平衡校验（feed 整个文档）：
      - handle_starttag：非 void 元素入栈
      - handle_endtag：void 元素（area/br/img/input/...）无需闭合；
        闭标签与栈顶匹配则弹出；非法嵌套则弹出至匹配项并记录中间未闭合标签；
        多余闭标签 → 错误
   d. 校验结束：栈非空（有标签未闭合）或 errors 非空 → FAIL
```

> **为何改用解析器标签栈**：替代旧的「双引号数量为奇数即报错」字符串级检查——文本内容 / JS 里的引号不再被误判，且能真正抓到「标签未闭合 / 非法嵌套 / 多余闭标签」。

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `check_html_only` | true | 仅检查 HTML 文件，无 HTML 时自动跳过 |

**判定标准**：全部通过 → score=1.0 PASS；任一有问题 → score=0.0 FAIL

## 三、常识约束指标（3 项）— HARD_SCORE

> **通过条件**：全部 3 个评估器均 PASS → S_common = +1；任一 FAIL → S_common = 0。
> **HARD_SCORE 失败不中断**：阶段内其他评估器继续执行，不阻塞后续质量评估阶段。
> **知识库**：`assets/knowledge/*.yaml`（详见 §六）

---

### 3.1 commonsense.info_accuracy — 知识准确性检查

**指标定义**：验证文档内容中的科学事实、数值常数、数学等式是否正确。是系统中最复杂的评估器，采用**三层检查架构**。

**检查算法**：

#### Phase 1：内置自动检查（始终执行）

| 检查项 | 方法 | 说明 |
|--------|------|------|
| **算术等式验证** | `_check_arithmetic()` | 正则提取完整左侧多项式等式，`_eval_simple_expr()` 求值（支持 */优先于+-）；支持带余除法 `A/B=C余D` 验证 `C*B+D==A`；竖式计算上下文跳过（前后 40 字符内含"进位""退位"等关键词） |
| **常数校验** | `_check_constants()` | 从知识库加载 constants 列表，每个常数有 `extract_pattern`(正则)、`value`(标准值)、`tolerance`(容差)；提取文档中匹配值，偏差超容差则报告 |
| **误解检测** | `_check_misconceptions()` | 从知识库加载 misconceptions 列表，每条有 `pattern`(正则)、`correct`(正确表述)、`severity`("error"/"warning")；不计入 checks 计数 |

#### Phase 2：可配置规则检查（`fact_rules` 参数）

| 规则类型 | 说明 |
|---------|------|
| `must_contain` | 关键词必须出现在全局文本中 |
| `must_not_contain` | 错误表述不应出现 |
| `value_range` | 正则提取数值，检查 [min, max] |
| `regex_match` | 正则匹配/不匹配 |
| `number_in_context` | 关键词附近 context_chars(默认50) 范围内的数值检查 |
| `forbidden_pattern` | 严格禁止的模式 |

#### Phase 3：LLM 语义验证（可选，与规则解耦）

当 `judge_orchestrator` 可用时启用，使用 `info_accuracy` 模板。**LLM 独立评估原文事实性**——只传入 `content` / `title` / `subject`，**不注入 Phase 1-2 的规则可疑条目**（解耦）。对照实验表明：把规则误报（如把"报告字数 800"误读为"水的 pH 800"）作为"可疑条目"喂给 LLM 会严重污染整体评分。规则 findings 的误报过滤由 Phase 2.5 独立完成，与 LLM 语义分互不干扰。LLM 调用失败则回退到 Phase 1-2 结果。

#### Phase 2.5：fact_verdict 规则误报二次确认

规则 Phase 1-2 标记的 `error` findings 在触发一票否决前，先经 LLM 二次确认（`fact_verdict` 模板），逐条裁定是否为真错误：

- **分批调用**：候选 findings 按 `fact_verdict_batch_size`（默认 `FACT_VERDICT_BATCH_SIZE = 20`）分批，避免单次 prompt 过大导致 LLM 调用失败；每批独立调用，单批失败时该批默认保留（召回优先，不漏报）。
- **裁定规则**：LLM 返回 `is_real_error`；被判误报（`false`）的 finding 不计入 `rule_errors`（不触发一票否决），但仍写审计字段 `_llm_confirmed` / `_llm_reason`。
- **缺裁定默认真**：未拿到裁定的 finding 默认按真错误处理（召回优先）。

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `subjects` | 全部 | 知识库学科过滤 |
| `fact_rules` | [] | Phase 2 可配置规则列表 |
| `pass_threshold` | 0.8 | 通过阈值 |
| `max_file_chars` | 4000 | Phase 1-2 / LLM 每文件截断字符数 |
| `max_content_chars` | 8000 | Phase 1-2 合并文本总截断字符数 |
| `fact_verdict_batch_size` | 20 | Phase 2.5 fact_verdict 分批大小 |
| `llm_provider` | 默认 | LLM Provider 名称 |

> LLM Phase 3 传入 `info_accuracy` 模板的 `content` 另按 `llm_judge_combined_content_chars`（默认 6000）截断，与上述规则 Phase 截断参数相互独立。

**计分逻辑**：
- 纯 Rule-based：`raw_score = (checks_total - errors) / checks_total`；`passed = raw_score >= threshold`
- LLM 模式：LLM 各维度加权平均 / 10 归一化为 `avg_score`；通过条件 = `avg_score >= threshold` **且** 经 fact_verdict 确认后的 `rule_errors` 为空（双重门控）

**reason 生成**（面向用户可读，替代"N 处错误"模板）：
- 拼接各 LLM 维度分数（如 `factual_correctness=8.0, statement_accuracy=9.0`）；
- 若存在经确认的错误：列出具体错误描述（`_llm_reason` 优先，回退规则 `message`），最多 5 条，超出标注"共 N 处"；
- 若规则标记了疑似错误但经 fact_verdict 全部不成立：注明"规则标记 N 处疑似错误经 LLM 二次确认均不成立"。

**LLM 模板**：`assets/prompts/info_accuracy.yaml`（2 维度：factual_correctness 0.6、statement_accuracy 0.4）；二次确认用 `assets/prompts/fact_verdict.yaml`。

**知识库依赖**：`assets/knowledge/_defaults.yaml` + 对应学科文件

### 3.3 commonsense.logical_consistency — 逻辑一致性检查

**指标定义**：检查文档内容是否存在自相矛盾的陈述。支持 LLM 模式和 Rule-based 降级两种方式。

#### LLM 模式（正常模式）

当 `judge_orchestrator` 可用时，使用 `logical_consistency` 模板进行 LLM 评估：

- **模板**：`assets/prompts/logical_consistency.yaml`
- **维度**：internal_consistency(0.5)、causal_logic(0.3)、classification_logic(0.2)
- **通过条件**：各维度加权平均 / 10 ≥ 0.6

#### LLM 不可用降级行为

- **未配置 LLM**（orchestrator 为空）→ **SKIP**（score=0.0，不计入得分，避免 CPR 虚高）。
- **LLM 调用异常**（已配置但调用失败）→ 降级 Rule-based 具名变量赋值矛盾检测：

```
1. 正则提取具名变量赋值：([a-zA-Z_一-鿿][\w一-鿿]{0,19})\s*[=＝]\s*(\d+\.?\d*)
2. 跳过单字母 ASCII 变量（数学公式中的 x=5 等）
3. Per-file 检查，不同文件的变量互不比较
4. Pass 1：同名变量有多个不同值 → 矛盾
5. Pass 2：中文公共后缀匹配（如"有学生数"与"但学生数"共享后缀"学生数"）
```

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_content_chars` | 8000 | LLM 模式截断字符数 |
| `llm_provider` | 默认 | LLM Provider 名称 |

**判定标准**：无矛盾 → score=1.0 PASS；存在矛盾 → score=0.0 FAIL

## 四、质量评估指标（5 项）— LLM-as-Judge

> **评估方法**：全部通过 LLM-as-Judge 实现。
> **降级机制**：当 LLM 不可用时（未配置 `llm_config.yaml` / 鉴权失败 / 限流 / 连接失败 / 额度耗尽），降级为 **SKIP（score=0.0，不计入得分）**，避免默认 PASS 令 Reward 虚高。
> **稳定性控制**：每个模板 temperature=0.0, seed=42, num_samples=3；3 次独立采样取中位数；标准差 > 1.5 标记为"低置信度"。

### 4.1 LLM Judge 通用流程

所有 5 个 LLM Judge 评估器继承自 `BaseLLMJudgeEvaluator`，遵循统一流程：

```
1. 检查 judge_orchestrator 和 evidence_dir 是否可用
   不可用 / 额度耗尽（熔断）→ SKIP（score=0.0，不计入得分）

2. 收集文档内容，截断至 max_content_chars（默认 8000）

3. 子类通过 _build_variables() 构建模板变量

4. 调用 orchestrator.judge(
     constraint_id=evaluator_id,
     template_id=模板ID,
     variables=模板变量,
     evidence_dir=证据目录,
   )

5. 计算加权分数：
   weighted_score = sum(score * weight for each dim) / total_weight
   normalized = weighted_score / 10.0（归一化到 [0, 1]）

6. 通过条件：normalized >= 0.4 → PASS；否则 FAIL
```

**LLM Judge 降级行为**：

| 场景 | score | status |
|------|-------|--------|
| 未配置 LLM | 0.0 | SKIP（不计分） |
| 鉴权失败 / 网络 / 限流 | 0.0 | SKIP（不计分） |
| 额度耗尽（`LLMQuotaExceeded`） | 0.0 | SKIP + 同 sample 熔断（不计分） |
| 模板渲染/解析异常 | 0.0 | ERROR |

| `completeness` | 完整性 | 0.4 | [0, 10] |
| `relevance` | 相关性 | 0.3 | [0, 10] |
| `quality` | 质量 | 0.3 | [0, 10] |

**特有逻辑**：将原始需求 `task.input` 作为 `original_request` 和 `expected_output` 传入模板。

**聚合权重**：`pref.request_fulfillment` 在 PREFERENCE 层中占 0.34

## 五、评分计算方法

### 5.1 单项评分归一化

| 评估方法 | 原始分数 | 归一化 | 说明 |
|---------|---------|--------|------|
| RULE | 0.0 或 1.0 | 直接使用 | 通过/失败二值 |
| FACT_VERIFY | 0.0 或 1.0 | 直接使用 | 规则 findings 经 fact_verdict 确认 + LLM 语义分，双重门控 |
| LLM_CONSISTENCY | 0.0 或 1.0 | 直接使用 | LLM 加权平均 / 10 后与阈值比较 |
| LLM_JUDGE | [0, 10] | score / 10 → [0, 1] | 各维度加权平均后归一化 |
| VISION | [0, 10] | score / 10 → [0, 1] | 截图 + LLM 多模态评估（opt-in） |

### 5.2 Reward 公式

```
Reward = (S_format + S_common + w3 × S_soft + w4 × S_pref) / (1 + 1 + w3 + w4)
```

> **归一化设计（行业最佳实践）**：各维度 ∈ [0,1]，Reward 归一化到 **[0,1]、无负值**。
> format 失败的严重性由 **DR（格式通过率）+ fail-fast 短路** 体现，不再用负值惩罚——
> 避免异常值（如旧 -3）拉偏指标分布与可视化坐标轴。

| 符号 | 含义 | 取值范围 | 默认乘数 |
|------|------|---------|---------|
| S_format | 格式约束得分 | {0, +1} | — |
| S_common | 常识约束得分 | {0, +1} | — |
| S_soft | 软约束得分 | [0, 1] | w3 = 1.0 |
| S_pref | 偏好约束得分 | [0, 1] | w4 = 1.0 |

**Reward 范围**：[0.0, 1.0]（format 失败 fail-fast → 0.0；全满分 → 1.0）

### 5.3 各阶段得分计算

#### S_format（格式约束）

| 条件 | S_format |
|------|----------|
| format 阶段全部 PASS | +1 |
| format 阶段任一 FAIL | 0 |
| format 阶段 SKIP（前置失败） | 0 |

#### S_common（常识约束）

| 条件 | S_common |
|------|----------|
| commonsense 阶段全部 PASS | +1 |
| commonsense 阶段任一 FAIL | 0 |
| commonsense 阶段 SKIP | 0 |

#### S_soft（软约束）— 加权归一化

```
S_soft = Σ(weight_i × score_i) / Σ(weight_i)

默认权重：
  soft.teaching_logic:  0.5
  soft.content_diversity: 0.5
```

#### S_pref（偏好约束）— 加权归一化

```
S_pref = Σ(weight_i × score_i) / Σ(weight_i)

默认权重：
  pref.style_preference:     0.33
  pref.depth_preference:     0.33
  pref.request_fulfillment:  0.34
```

### 5.4 聚合指标

对一批样本的评估结果，`MetricsCalculator` 计算以下聚合指标：

| 指标 | 全称 | 计算方法 | 默认阈值 |
|------|------|---------|---------|
| **DR** | Delivery Rate（交付率） | 格式阶段 gate_passed 的样本数 / 总样本数 | ≥ 0.95 |
| **CPR** | Constraint Pass Rate（约束通过率） | 格式+常识均 gate_passed 的样本数 / 总样本数 | ≥ 0.90 |
| **Reward** | 全样本平均 Reward | Σ(reward) / 总样本数 | ≥ 0.70 |
| **avg_soft** | 平均内容质量分 | Σ(s_soft) / 总样本数（独立指标，不混入 Reward） | — |
| **avg_pref** | 平均用户偏好分 | Σ(s_pref) / 总样本数（独立指标，不混入 Reward） | — |
| **CondR** | Conditional Reward（条件奖励） | 通过双门控样本的 Reward 均值（无通过样本时为 0） | — |
| **Time** | 平均完成时间 | Σ(total_duration_ms) / 总样本数 | — |
| **llm_skipped** | LLM 跳过样本数 | LLM 不可用而 SKIP 的样本计数（用于判断 LLM 评估覆盖率） | — |
| **failure_breakdown** | 失败分类统计 | 各 constraint_id 的 FAIL 次数计数 | — |
