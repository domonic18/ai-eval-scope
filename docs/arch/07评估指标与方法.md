# 评估指标与方法

> 本文档完整描述系统的评估指标体系，包括全部 14 个评估器的指标定义、评估方法（Rule-based / LLM-as-Judge）、检查算法、评分计算方法，以及相关配置文件的维护位置。属于 [04 评估引擎设计](./04评估引擎设计.md) 的详细展开。

---

## 一、总览

### 1.1 评估体系概览

系统采用**三阶段级联**评估架构，包含 **14 个评估器**，分为四个约束层级：

| 阶段 | 评估器数 | 层级 | 失败行为 | 评估方法 |
|------|---------|------|---------|---------|
| **Stage 1: format** | 4 | HARD_GATE | 任一失败 → S_format = -3，后续阶段全部 SKIP | Rule-based |
| **Stage 2: commonsense** | 5 | HARD_SCORE | 任一失败 → S_common = 0，不中断阶段内其他评估器，不阻塞后续阶段 | Rule + LLM |
| **Stage 3: quality** | 5 | SOFT(2) + PREFERENCE(3) | 归一化 [0,1] 加权平均，不影响门控 | LLM-as-Judge |

**完整评估器列表**：

| # | evaluator_id | 名称 | 阶段 | 层级 | 方法 |
|---|---|---|---|---|---|
| 1 | `format.response_format` | 文件格式检查 | format | HARD_GATE | RULE |
| 2 | `format.html_validity` | HTML 有效性检查 | format | HARD_GATE | RULE |
| 3 | `commonsense.info_accuracy` | 知识准确性检查 | commonsense | HARD_SCORE | FACT_VERIFY |
| 4 | `commonsense.chronological_order` | 时序正确性检查 | commonsense | HARD_SCORE | RULE |
| 5 | `commonsense.logical_consistency` | 逻辑一致性检查 | commonsense | HARD_SCORE | LLM_CONSISTENCY |
| 6 | `commonsense.math_formula` | 公式正确性检查 | commonsense | HARD_SCORE | MATH_VERIFY |
| 7 | `commonsense.unit_consistency` | 单位一致性检查 | commonsense | HARD_SCORE | RULE |
| 8 | `soft.teaching_logic` | 教学逻辑 | quality | SOFT | LLM_JUDGE |
| 9 | `soft.content_diversity` | 内容多样性 | quality | SOFT | LLM_JUDGE |
| 10 | `pref.style_preference` | 风格偏好 | quality | PREFERENCE | LLM_JUDGE |
| 11 | `pref.depth_preference` | 深度偏好 | quality | PREFERENCE | LLM_JUDGE |
| 12 | `pref.request_fulfillment` | 需求满足度 | quality | PREFERENCE | LLM_JUDGE |

### 1.2 评估方法分类

| 方法标识 | 说明 | 使用的评估器 |
|---------|------|-------------|
| `RULE` | 纯规则匹配（正则、文件属性检查），无需 LLM | format 全部、chronological_order、unit_consistency |
| `FACT_VERIFY` | 事实性验证（知识库匹配 + 可选 LLM 语义验证） | info_accuracy |
| `MATH_VERIFY` | 数学公式验证（括号匹配 + 算术等式 + 符号公式） | math_formula |
| `LLM_CONSISTENCY` | 逻辑一致性验证（LLM 模式为主，Rule-based 降级） | logical_consistency |
| `LLM_JUDGE` | 纯 LLM 评估（多次采样取中位数） | quality 阶段全部 5 个 |

### 1.3 公共基础设施

**样本路径解析**：所有评估器通过 `_get_output_dir(sample)` 获取输出目录，支持 Path / dict / 有 `output_dir` 属性的对象。

**文本内容收集**：Commonsense 和 Quality 阶段的评估器使用 `_collect_text_content(output_dir)` 或 `_collect_file_texts(output_dir)` 将文档内容合并。遍历 `*.md, *.html, *.htm` 文件，HTML 文件去除标签后以换行拼接。

## 二、格式约束指标（2 项）— HARD_GATE

> **评估方法**：全部为 RULE，无需 LLM。
> **通过条件**：全部 4 个评估器均 PASS → S_format = +1；任一 FAIL → S_format = -3。
> **配置文件**：`assets/rules/default_rule_set.yaml`

---

### 2.1 format.response_format — 文件格式检查

**指标定义**：验证所有输出文件的扩展名和内容头是否为有效的 Markdown 或 HTML。

**检查算法**：

```
1. 从 params["allowed_formats"] 获取允许格式列表（默认 ["md", "html"]）
2. 构建扩展名映射：md→[".md",".markdown"], html→[".html",".htm"]
3. rglob("*") 遍历 output_dir 下所有文件（排除 _manifest.json）
4. 对每个文件：
   a. 扩展名匹配 → 进入内容头校验：
      - .md/.markdown：前 500 字符非空即通过
      - .html/.htm：前 500 字符以 "<" 开头 或 包含 "<!doctype" 或 "<html"
   b. 扩展名不匹配 → 标记无效
```

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `allowed_formats` | `["md", "html"]` | 允许的文件格式列表 |

**判定标准**：全部通过 → score=1.0 PASS；任一不通过 → score=0.0 FAIL

**已知局限**：Markdown 内容头检查过于宽松（仅判断非空）；不检查文件编码。

---

### 2.2 format.html_validity — HTML 有效性检查

**指标定义**：验证 HTML 文件标签是否闭合、引号是否匹配。

**检查算法**：

```
1. 收集 .html/.htm 文件
2. check_html_only=True 且无 HTML 文件 → 自动 PASS
3. 对每个文件：
   a. 空文件检查：content.strip() 为空 → FAIL
   b. HTML 结构：以 "<" 开头 或 包含 "<!doctype" 或 "<html"
   c. 关键标签闭合：<html>/<head>/<body> 有开标签但无闭标签 → 错误
   d. 引号匹配：双引号数量为奇数 → 错误
```

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `check_html_only` | true | 仅检查 HTML 文件，无 HTML 时自动跳过 |

**判定标准**：全部通过 → score=1.0 PASS；任一有问题 → score=0.0 FAIL

## 三、常识约束指标（5 项）— HARD_SCORE

> **通过条件**：全部 5 个评估器均 PASS → S_common = +1；任一 FAIL → S_common = 0。
> **HARD_SCORE 失败不中断**：阶段内其他评估器继续执行，不阻塞后续质量评估阶段。
> **知识库**：`assets/knowledge/*.yaml`（详见 §六）

---

### 3.1 commonsense.info_accuracy — 知识准确性检查

**指标定义**：验证文档内容中的科学事实、数值常数、数学等式是否正确。是系统中最复杂的评估器，采用**三层检查架构**。

**检查算法**：

#### Phase 1：内置自动检查（始终执行）

| 检查项 | 方法 | 说明 |
|--------|------|------|
| **算术等式验证** | `_check_arithmetic()` | 正则提取完整左侧多项式等式，`_eval_simple_expr()` 求值（支持 */优先于+-）；支持带余除法 `A/B=C余D` 验证 `C*B+D==A`；竖式计算上下文跳过（前后 40 字符内含"进位""退位"等关键词） |
| **常数校验** | `_check_constants()` | 从知识库加载 constants 列表，每个常数有 `extract_pattern`(正则)、`value`(标准值)、`tolerance`(容差)；提取文档中匹配值，偏差超容差则报告 |
| **误解检测** | `_check_misconceptions()` | 从知识库加载 misconceptions 列表，每条有 `pattern`(正则)、`correct`(正确表述)、`severity`("error"/"warning")；不计入 checks 计数 |

#### Phase 2：可配置规则检查（`fact_rules` 参数）

| 规则类型 | 说明 |
|---------|------|
| `must_contain` | 关键词必须出现在全局文本中 |
| `must_not_contain` | 错误表述不应出现 |
| `value_range` | 正则提取数值，检查 [min, max] |
| `regex_match` | 正则匹配/不匹配 |
| `number_in_context` | 关键词附近 context_chars(默认50) 范围内的数值检查 |
| `forbidden_pattern` | 严格禁止的模式 |

#### Phase 3：LLM 语义验证（可选）

当 `judge_orchestrator` 可用时启用，使用 `info_accuracy` 模板，将 Phase 1-2 的 warning/error 传入 LLM 供重点验证。LLM 调用失败则回退到 Phase 1-2 结果。

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `subjects` | 全部 | 知识库学科过滤 |
| `fact_rules` | [] | Phase 2 可配置规则列表 |
| `pass_threshold` | 0.8 | 通过阈值 |
| `max_file_chars` | 4000 | LLM Phase 每文件截断字符数 |
| `max_content_chars` | 8000 | LLM Phase 总内容截断字符数 |
| `llm_provider` | 默认 | LLM Provider 名称 |

**计分逻辑**：
- 纯 Rule-based：`raw_score = (checks_total - errors) / checks_total`；`passed = raw_score >= threshold`
- LLM 模式：各维度加权平均 / 10 归一化；需 rule-based 无 error + LLM 分数 >= threshold 才 PASS

**LLM 模板**：`assets/prompts/info_accuracy.yaml`（2 个维度：factual_correctness 0.6, statement_accuracy 0.4）

**知识库依赖**：`assets/knowledge/_defaults.yaml` + 对应学科文件

### 3.2 commonsense.chronological_order — 时序正确性检查

**指标定义**：检查文档中的历史年份和序号是否按合理顺序排列。

**检查算法**：

```
1. 合并所有文档文本内容
2. 提取年份：(?:公元)?(\d{3,4})\s*年，过滤 0 < year ≤ 2100
3. 提取序号：第([一二三四五六七八九十百千\d]+)[章节步骤期]
4. 当前：检测到年份回溯时不报告（pass），序号只统计数量
```

**判定标准**：当前始终 score=1.0 PASS（为空实现，时序回溯暂不报告）

**已知局限**：不检查序号递增性；年份提取不区分历史年份和数学数字。

---

### 3.3 commonsense.logical_consistency — 逻辑一致性检查

**指标定义**：检查文档内容是否存在自相矛盾的陈述。支持 LLM 模式和 Rule-based 降级两种方式。

#### LLM 模式（正常模式）

当 `judge_orchestrator` 可用时，使用 `logical_consistency` 模板进行 LLM 评估：

- **模板**：`assets/prompts/logical_consistency.yaml`
- **维度**：internal_consistency(0.5)、causal_logic(0.3)、classification_logic(0.2)
- **通过条件**：各维度加权平均 / 10 ≥ 0.6

#### Rule-based 降级模式

LLM 不可用时，执行具名变量赋值矛盾检测：

```
1. 正则提取具名变量赋值：([a-zA-Z_一-鿿][\w一-鿿]{0,19})\s*[=＝]\s*(\d+\.?\d*)
2. 跳过单字母 ASCII 变量（数学公式中的 x=5 等）
3. Per-file 检查，不同文件的变量互不比较
4. Pass 1：同名变量有多个不同值 → 矛盾
5. Pass 2：中文公共后缀匹配（如"有学生数"与"但学生数"共享后缀"学生数"）
```

**配置参数**：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_content_chars` | 8000 | LLM 模式截断字符数 |
| `llm_provider` | 默认 | LLM Provider 名称 |

**判定标准**：无矛盾 → score=1.0 PASS；存在矛盾 → score=0.0 FAIL

---

### 3.4 commonsense.math_formula — 公式正确性检查

**指标定义**：验证文档中的数学公式是否正确，覆盖 LaTeX 括号匹配、算术等式验证和符号公式校验。

**检查算法**（per-file）：

```
1. LaTeX 括号匹配
   正则提取 $...$ 公式，检查 () 和 {} 成对

2. 算术等式验证（复用 InfoAccuracyEvaluator 的正则和求值器）
   - 完整左侧多项式等式：提取并求值
   - 带余数除法：验证 C*B + D == A
   - 含竖式计算上下文跳过

3. 符号公式校验（_check_symbolic_formulas）
   a. 从知识库加载 domain_facts.math_formulas（assets/knowledge/math.yaml）
   b. formula_normalizer.build_formula_index() 构建归一化公式索引
   c. 正则提取文档中的公式声明：
      - "长方形面积=长x宽" 形式
      - "圆面积公式：..." 形式
   d. normalize_formula_name() 去除"的"字
   e. normalize_formula() 将 Unicode 数学符号归一化为 ASCII（×→*, ÷→/, ²→^2 等）
   f. 与知识库已知公式匹配（精确匹配优先，其次子串匹配）
```

**公式归一化器**（`agent_eval/evaluation/evaluators/formula_normalizer.py`）：

| 函数 | 说明 |
|------|------|
| `normalize_formula(formula)` | Unicode 数学符号 → ASCII（×→*, ÷→/, ²→^2, π→pi 等） |
| `normalize_formula_name(name)` | 去除中文"的"字 |
| `build_formula_index(facts)` | 从 domain_facts 构建归一化公式查找索引 |

**判定标准**：无错误 → score=1.0 PASS；有错误 → score=0.0 FAIL

**知识库依赖**：`assets/knowledge/math.yaml`（平面图形 9 条、立体图形 9 条、应用公式 7 条、代数公式 5 条）

---

### 3.5 commonsense.unit_consistency — 单位一致性检查

**指标定义**：检查文档中物理量单位使用是否正确。

**检查算法**：

```
1. 合并所有文档文本内容
2. 密度单位：密度.*?(\d+\.?\d*)\s*kg/m(?!²|³) → 密度单位可能错误
3. 重力加速度单位：g\s*[=≈]\s*(\d+\.?\d*)\s*m/s(?![²³]) → 单位可能错误
```

**判定标准**：无错误 → score=1.0 PASS；检测到问题 → score=0.0 FAIL

**已知局限**：仅覆盖密度和重力加速度两个场景。

## 四、质量评估指标（5 项）— LLM-as-Judge

> **评估方法**：全部通过 LLM-as-Judge 实现。
> **降级机制**：当 LLM 不可用时（未配置 `llm_config.yaml` 或 Provider 连接失败），自动降级为 score=0.7 PASS。
> **稳定性控制**：每个模板 temperature=0.0, seed=42, num_samples=3；3 次独立采样取中位数；标准差 > 1.5 标记为"低置信度"。

### 4.1 LLM Judge 通用流程

所有 5 个 LLM Judge 评估器继承自 `BaseLLMJudgeEvaluator`，遵循统一流程：

```
1. 检查 judge_orchestrator 和 evidence_dir 是否可用
   不可用 → 降级模式：返回 score=0.7, PASS

2. 收集文档内容，截断至 max_content_chars（默认 8000）

3. 子类通过 _build_variables() 构建模板变量

4. 调用 orchestrator.judge(
     constraint_id=evaluator_id,
     template_id=模板ID,
     variables=模板变量,
     evidence_dir=证据目录,
   )

5. 计算加权分数：
   weighted_score = sum(score * weight for each dim) / total_weight
   normalized = weighted_score / 10.0（归一化到 [0, 1]）

6. 通过条件：normalized >= 0.4 → PASS；否则 FAIL
```

**LLM Judge 降级行为**：

| 场景 | score | status |
|------|-------|--------|
| 未配置 LLM | 0.7 | PASS |
| Provider 连接失败 | 0.7 | PASS |
| 模板渲染/解析异常 | 0.0 | ERROR |

## 五、评分计算方法

### 5.1 单项评分归一化

| 评估方法 | 原始分数 | 归一化 | 说明 |
|---------|---------|--------|------|
| RULE | 0.0 或 1.0 | 直接使用 | 通过/失败二值 |
| FACT_VERIFY | 0.0 或 1.0 | 直接使用 | 基于 raw_score 与 threshold 比较 |
| MATH_VERIFY | 0.0 或 1.0 | 直接使用 | 通过/失败二值 |
| LLM_CONSISTENCY | 0.0 或 1.0 | 直接使用 | LLM 加权平均 / 10 后与阈值比较 |
| LLM_JUDGE | [0, 10] | score / 10 → [0, 1] | 各维度加权平均后归一化 |

### 5.2 Reward 公式

```
Reward = S_format + S_common + w3 × S_soft + w4 × S_pref
```

| 符号 | 含义 | 取值范围 | 默认乘数 |
|------|------|---------|---------|
| S_format | 格式约束得分 | {-3, 0, +1} | — |
| S_common | 常识约束得分 | {0, +1} | — |
| S_soft | 软约束得分 | [0, 1] | w3 = 1.0 |
| S_pref | 偏好约束得分 | [0, 1] | w4 = 1.0 |

**理论 Reward 范围**：最差 -3.0（格式失败）→ 最佳 +4.0（全部满分）

### 5.3 各阶段得分计算

#### S_format（格式约束）

| 条件 | S_format |
|------|----------|
| format 阶段全部 PASS | +1 |
| format 阶段任一 FAIL | -3 |
| format 阶段 SKIP（前置失败） | 0 |

#### S_common（常识约束）

| 条件 | S_common |
|------|----------|
| commonsense 阶段全部 PASS | +1 |
| commonsense 阶段任一 FAIL | 0 |
| commonsense 阶段 SKIP | 0 |

#### S_soft（软约束）— 加权归一化

```
S_soft = Σ(weight_i × score_i) / Σ(weight_i)

默认权重：
  soft.teaching_logic:  0.5
  soft.content_diversity: 0.5
```

#### S_pref（偏好约束）— 加权归一化

```
S_pref = Σ(weight_i × score_i) / Σ(weight_i)

默认权重：
  pref.style_preference:     0.33
  pref.depth_preference:     0.33
  pref.request_fulfillment:  0.34
```

### 5.4 聚合指标

对一批样本的评估结果，`MetricsCalculator` 计算以下聚合指标：

| 指标 | 全称 | 计算方法 | 默认阈值 |
|------|------|---------|---------|
| **DR** | Delivery Rate（交付率） | 格式阶段 gate_passed 的样本数 / 总样本数 | ≥ 0.95 |
| **CPR** | Constraint Pass Rate（约束通过率） | 格式+常识均 gate_passed 的样本数 / 总样本数 | ≥ 0.90 |
| **Reward** | 全样本平均 Reward | Σ(reward) / 总样本数 | ≥ 0.70 |
| **CondR** | Conditional Reward（条件奖励） | 通过双门控样本的 Reward 均值（无通过样本时为 0） | — |
| **Time** | 平均完成时间 | Σ(total_duration_ms) / 总样本数 | — |
| **failure_breakdown** | 失败分类统计 | 各 constraint_id 的 FAIL 次数计数 | — |

### 6.2 Prompt 模板结构

```yaml
template_id: string          # 模板唯一标识

dimensions:                  # 评分维度列表

    name: string             # 维度名称
    description: string      # 维度说明

output_schema: dict          # 输出 JSON Schema
temperature: 0.0             # 采样温度
seed: 42                     # 随机种子
num_samples: 3               # 采样次数
```
