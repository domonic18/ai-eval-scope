## 一、总体策略

### 1.1 评估优先策略

采用 **"评估先行、执行后补"** 的开发策略：

```
阶段一：评估优先                          阶段二：自动化补全
┌──────────────────────────┐           ┌──────────────────────────┐
│ 手动获取 Agent 产出物     │           │ ExecutionAgent 自动化     │
│        ↓                  │           │        ↓                  │
│ 手动打包为 ExecutionPkg   │    ──→    │ Agent 端到端生成 Pkg      │
│        ↓                  │  演进     │        ↓                  │
│ agent-eval eval 评估      │           │ agent-eval pipeline 一步到位│
│        ↓                  │           │        ↓                  │
│ 输出报告 + 指标           │           │ 输出报告 + 指标           │
└──────────────────────────┘           └──────────────────────────┘
```

**核心思路**：
- 前期通过 `eval-only` 模式跑通评估链路，用户手动或简单脚本准备 Agent 产出物
- 提供轻量的 **PackageBuilder** 工具，将原始产出物（Markdown/HTML 文档集）快速打包为标准 ExecutionPackage
- 评估体系验证成熟后，再补全 ExecutionAgent（基于 claude-agent-sdk）实现自动化执行

**评估目标**：以 **Markdown/HTML 文档集**（多文档组合）为主要评估对象，暂不支持 PPTX。HTML 文档前期手动放置在工作目录，后期通过 API 从 Agent 自动获取。

**评估双模式**：
- **PipelineEngine（默认）**：固定级联管线，确定性高，无额外 token 消耗。通过 `--eval-mode=pipeline` 或省略参数触发
- **EvaluationAgent（可选）**：Agent 自适应评估，读取 Markdown 评测计划，用于复杂场景。需显式指定 `--eval-mode=agent`

                                                ↓
                 ┌── Sprint 7b (后端+数据模型) ──→ Sprint 7c (认证+多租户) ─┐
                 │                                                           │
                 └── Sprint 7b ──→ Sprint 7d (摄取服务) ──┐                  │
                                                              ↓                 ↓
                                Sprint 5 ──→ Sprint 7e (评估器对接) ──→ Sprint 7f (前端新建+Query API) ──→ Sprint 7g (运维收尾)
Sprint 1 ──→ Sprint 8 (ExecutionAgent) ──→ Sprint 9 (集成交付)            ↑ 可与 Sprint 8/9 并行

- Sprint 1 是所有后续迭代的基础
- Sprint 2 → Sprint 4 为评估核心链路，串行推进
- Sprint 5 贯通评估链路为用户可用形态，是 P1 阶段的交付里程碑
- Sprint 6（多模态）依赖 Sprint 3（Provider 抽象层），可与 Sprint 4/5 并行准备
- Sprint 7（规则管理）依赖 Sprint 2 + Sprint 4（评估器注册 + 完整链路）
- Sprint 7a（Web Portal MVP）依赖 Sprint 5（报告输出）和 Sprint 7（项目配置），也可在 Sprint 5 后提前启动
- **Sprint 7b（后端+数据模型）依赖 Sprint 7a，是可观测平台重构的起点**
- **Sprint 7c（认证+多租户）依赖 Sprint 7b；Sprint 7d（摄取服务）依赖 Sprint 7b**
- **Sprint 7e（评估器对接 ResultSink）依赖 Sprint 7d（摄取 API）与 Sprint 5（eval 流程）**
- **Sprint 7f（前端新建 + Query API）依赖 Sprint 7c + 7d；Sprint 7g（运维收尾）依赖 Sprint 7f**
- **Sprint 7 系列（7b–7g）与 Sprint 8/9（ExecutionAgent / 集成交付）无强依赖，可并行推进**
- Sprint 8（ExecutionAgent + SUT Tools）在评估体系成型后启动，依赖 Sprint 1（Agent 模块目录 + AgentConfig）
- Sprint 9（集成交付）依赖 Sprint 7 + Sprint 8

| 核心类型 | 枚举定义（RunMode、EvalStatus、ConstraintTier、EvalMethod、EvalMode）、公共异常 |

### Sprint 2：评估引擎（Rule-based）

**目标**：实现基于规则的评估器（格式约束 + 常识约束），完成级联控制与评分聚合，达到 **M1：Rule-based 评估可用**。

**范围**：

| 工作项 | 说明 |
|--------|------|
| EvaluatorRegistry | 装饰器注册模式、工厂创建 |
| BaseEvaluator ABC | 统一评估器接口 `evaluate(sample, context) → ConstraintResult` |
| 格式评估器（4 项） | 文件格式（MD/HTML）、文档数量、结构规范性、HTML 有效性 |
| 常识 Rule-based 评估器 | 时序正确性、单位一致性 |
| 常识 Fact-verify 评估器 | 知识准确性（对比知识库） |
| 常识 Math-verify 评估器 | 公式正确性检查 |
| PipelineStage | 单阶段执行器，门控判定 |
| CascadeController | 三阶段级联编排，短路控制（fail_fast / continue_all） |
| ScoreAggregator | Reward 公式实现（S_format + S_common + w3×S_soft + w4×S_pref） |
| MetricsCalculator | DR / CPR / Reward / CondR / Time 计算 |
| **最小 eval 流程** | **可在 Python 中直接调用评估引擎对 ExecutionPackage 执行评估（SDK 方式，无需 CLI）** |

**依赖**：Sprint 1

**交付物**：
- 格式门控 + 常识门控的 Rule-based 评估可用
- 级联控制正确（格式门控失败 → 后续阶段 SKIP）
- 评分聚合与指标计算输出正确
- **可通过 Python SDK 对 Sprint 1 准备的黄金样本执行评估并输出结果**

**验证标准**：
1. 合法文档集通过格式门控，S_format = +1
2. 格式不合法文档集被门控拦截，S_format = -3，后续阶段标记 SKIP
3. 常识检查失败时，S_common = 0
4. DR/CPR/Reward 计算结果与手工计算一致
5. 级联短路逻辑：Stage 1 失败后 Stage 2/3 不执行
6. 黄金样本回归测试全部通过
7. 评估器专项测试覆盖率 ≥ 90%

### Sprint 4：评估引擎（LLM Judge）

**目标**：接入 LLM Judge 评估器（软约束 + 偏好约束），实现完整的评估链路与溯源能力，达到 **M2：LLM 评估可用**。

**范围**：

| 工作项 | 说明 |
|--------|------|
| 常识 LLM 评估器 | 逻辑一致性（commonsense.logical_consistency） |
| 软约束 LLM 评估器 | 教学逻辑（soft.teaching_logic）、内容多样性（soft.content_diversity） |

### Sprint 8：ExecutionAgent 与 SUT Tools

**目标**：基于 claude-agent-sdk 实现 ExecutionAgent，端到端驱动评测执行流程；实现 SUT Tools（MCP）提供被测系统交互能力；支持 `agent-eval run` 模式。本迭代取代传统 HTTPDriver/CLIDriver 分层架构，采用 Agent-Driven 方案。

**范围**：

| 工作项 | 说明 |
|--------|------|
| **ExecutionAgent** | **基于 claude-agent-sdk 的执行 Agent：理解任务描述、智能构建 SUT 请求、自适应处理异常、采集结果、生成 ExecutionPackage** |
| **ExecutionAgent System Prompt** | **构建 Agent 的系统提示词：角色定义、职责说明、执行规则、输出规范** |
| **ExecutionAgent Task Prompt** | **构建任务提示词：从 Task 模型提取任务 ID、输入参数、预期结果、约束条件、目录模式信息** |
| **SUTToolServer** | **MCP Server 实现，注册所有 SUT Tools，转换为 MCP Server 实例供 Claude Agent SDK 使用** |
| **SUT Tool: invoke_http_sut** | **HTTP 方式调用被测 Agent（原 HTTPDriver 的逻辑下沉为此 Tool），支持请求模板、超时、重试** |
| **SUT Tool: invoke_cli_sut** | **CLI 方式执行被测 Agent（原 CLIDriver 的逻辑下沉为此 Tool），子进程管理** |
| **SUT Tool: scan_directory** | **目录扫描 Tool：遍历指定目录，收集文件信息，生成清单（Agent 可调用以了解产出物结构）** |
| **SUT Tool: read_file** | **文件读取 Tool：读取执行结果文件内容** |
| **SUT Tool: collect_results** | **结果收集 Tool：收集执行输出文件，归档到 workspace** |
| **SUT Tool: write_package** | **执行包写入 Tool：将收集的结果组装为 ExecutionPackage 并写入 workspace** |
| **SUT Tool: list_files** | **文件列表 Tool：列出指定目录下的文件** |
| **AgentSession** | **Agent 会话记录模型：保存完整交互历史（消息、token 用量、工具调用次数、耗时）** |
| **Agent 结构化日志** | **Agent 会话必须输出结构化日志（非自然语言）：tool_calls 列表、每步结果、执行时间、token 消耗、成本估算** |
| **BudgetController** | **预算控制器：基于 token 用量估算成本，max_budget_usd 硬性限制** |
| CLI `run` 命令 | 实现 `agent-eval run --task-set xxx --sut-config xxx`，通过 ExecutionAgent 驱动执行 |
| Orchestrator（run-only 模式） | 补全 run-only 模式编排：加载配置 → 初始化 ExecutionAgent → 执行任务集 → 写入 ExecutionPackage |
| Mock SUT Tools | 测试用 Mock SUTToolServer，返回模拟的 MD/HTML 文档集 |
| SUT 配置 | sut_config.yaml 配置文件格式定义与加载 |

**依赖**：Sprint 1（Agent 模块目录 + AgentConfig）+ Sprint 5（eval-only 已通，Orchestrator 骨架）

**交付物**：
- `agent-eval run` 可通过 ExecutionAgent 自动驱动 SUT 执行任务
- ExecutionAgent 理解任务描述，智能调用 SUT Tools
- SUT Tools（MCP）提供完整的被测系统交互能力（HTTP/CLI/文件操作）
- Agent 会话输出结构化日志（tool_calls、results、timing、cost）
- 手动打包（PackageBuilder）与 ExecutionAgent 产出相同的 ExecutionPackage 格式
- Mock SUT Tools 可用于无真实 SUT 的集成测试

**验证标准**：
1. ExecutionAgent 接收 Task 后，可自主调用 SUT Tools 完成执行（至少 invoke_http_sut + collect_results + write_package）
2. `agent-eval run` 通过 ExecutionAgent 执行任务集，生成标准 ExecutionPackage
3. 自动生成的 ExecutionPackage 与手动 PackageBuilder 打包的结构一致
4. Agent 会话日志为结构化格式：包含 tool_calls（工具名、参数、结果）、timing（开始/结束时间）、token 用量、成本估算
5. SUTToolServer 可转换为 MCP Server 实例，工具列表完整（7 个工具全部注册）
6. BudgetController 在超出预算时正确终止 Agent 执行
7. Mock SUT Tools 下集成测试通过：从 `agent-eval run` 到 ExecutionPackage 生成
8. 执行失败时，Agent 会话日志正确记录错误信息和已执行的步骤
9. 单元测试 + 集成测试（Mock SUT Tools）通过

**参考文档**：
- [03执行引擎设计](../arch/03执行引擎设计.md) §三（ExecutionAgent）、§四（SUT Tools MCP）、§五（Agent 会话管理与预算控制）

| 风险 | 影响 | 应对策略 |
